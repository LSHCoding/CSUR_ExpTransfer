## §6.2 Parametric-to-Tokenized Transformation（P7）

P7 把已内化在参数化模型权重中的经验外化为可被其他系统消费的 Token-level artifacts。源端是 Policy 或 Evaluator 参数（承载这些参数的模型可由任意 modality 实现，LLM、VLM、VLA 均可），目标端是 agent trajectories、tool-use demonstrations、GUI action traces、step-level labels、critiques、preference annotations 等 Tokenized 载体。P7 与 P5（Tokenized → Policy）方向相反：P5 把离散经验写入参数，P7 从参数中读出可复用的经验表达。

P7 仍是一条 representation-to-representation pathway。外化出的 trajectory 并非凭空产生，它重新编码了曾经经由 P4/P5 写入参数的经验语义，满足 §2.3 的经验语义锚定与经验内容承载两个条件。与 P1–P6 不同的是 source 端的形态。P3 的 source 是一条确定的 trajectory，转化算子是确定性的 re-encoding；P7 的 source 是参数这种聚合态表示，贡献它的多条原始 trajectory 已在权重分布中彼此叠加、无法再被个别定位，转化算子相应地包含一次采样——$\mathcal{T}$ 从 $p(e \mid \text{prompt}, \text{env})$ 中抽取样本，而非对某个给定的 $e$ 做重编码。

这一 source 性质直接决定了 P7 整节的两个组织性事实。采样有方差，低质量样本必须由 verifier / filter 在事后剔除，这解释了为何 P7 方法几乎无一例外地挂载验证环节。采样可重复进行并保留最优样本，这带来 best-of-N 效应：在筛选后的 teacher 数据上训练的 student，其可靠性可以高于 teacher 单次采样的平均水平。

P7 需与三类相邻过程区分。普通 inference 中模型生成的 CoT 或答案只服务于当前 query，没有独立归档与复用意图，不构成 P7。Self-training 与 iterative bootstrapping 中模型生成的 rollouts 被同一训练循环的后继版本直接消费，更接近 composite pipeline（§7）。与 P1/P2 的区分可归结为一条可操作判据：追溯转化的输入端——若输入是一条已存在的 trajectory、转化是对这条 trajectory 的重表述，归 P1/P2；若输入只是 prompt 或 task specification、被外化的内容来自模型已内化的能力，归 P7。

按外化经验在下游系统中的消费角色，P7 分为两类：**Demonstration Externalization** 外化“如何行动”的经验，来自 Policy 参数；**Evaluative Supervision Externalization** 外化“如何评价”的经验，来自 Evaluator 参数。两类的分界线在于 Evaluator 判断的去向——判断被物化为独立、可复用的监督数据则归入后者，判断只作为内部过滤开关则仍是前者内部的质量控制机制，不单独成类。

### §6.2.1 Demonstration Externalization

Demonstration Externalization 指参数化 Policy 或 teacher agent 将其隐式决策能力外化为可供 student 模仿学习的行为示范。目标 artifact 通常是 agent trajectories、tool-use dialogues、web/GUI/mobile interaction traces 或 reasoning-action paths，下游 consumer 通过 SFT、behavioral cloning、imitation-style training，或将其作为 RL 的初始化数据来吸收这些示范。消费方式是 SFT 还是 RL 不改变 artifact 的载体属性，归类依据是被外化 artifact 的形态，而非它如何被消费。

Demonstration Externalization 的根本定性约束是能力天花板：student 在任务能力维度上继承的是 teacher 的上界，P7 用 teacher 的一次性推理成本换取的是 student 的低推理成本、可规模化与领域专用化，而非超出 teacher 的能力。前述 best-of-N 效应能把 student 的可靠性抬高到 teacher 平均水平之上，但抬高的是同一能力前沿内的稳定性，而非能力前沿本身。

本文以**环境合成度**作为 §6.2.1 的组织轴。它锚定在 §2.1 四元组的 observation $o$ 上：$o$ 越是来自真实环境的客观返回，外化轨迹的 grounding 越强；$o$ 越是由模型自身生成，轨迹越接近一次纯参数读出。沿这条轴，pseudo-success 与 hallucinated observation 的风险递增，verifier 的必要性也相应递增。

**真实环境锚定（real-environment-grounded）。** Teacher 在真实网站、真实 MCP server 或真实操作系统上交互，$o$ 是环境返回的客观后果。Explorer [Pah25] 让 GPT-4o 在真实网页中以 exploration-driven 方式自主发现任务、执行多步交互，经 proposer / refiner / summarizer / verifier 流水线产出带页面状态、动作序列与任务总结的 multimodal trajectories。InSTA [Tra25] 将这一思路推至 internet scale，使强模型在大规模真实网站上自动提出任务、执行操作并生成 reasoning traces，再以 judge-based filtering 只保留高质量成功轨迹。AutoSurfer [Fai26] 先系统性 surf 真实网站、识别功能空间，再整理为含页面状态、动作历史与推理的 task tuples。SynthAgent [Wan25ar] 在目标网站上先做 element-level exploration 与任务合成，再于收集后全局 refine，产出更贴合特定站点分布的 synthetic supervision。Structured Distillation of Web Agent Capabilities [Lu26] 以 Agent-as-Annotators 框架让 Gemini teacher 生成带结构化 reasoning block 的成功轨迹。计算机与移动端方面，Fara-7B [Awa25] 经 FaraGen 产出 screenshot-thought-action trajectories 并由三重 verifier 筛选；OpenMobile [Che26d] 以 expert 接管修正 learner 偏差，把 corrected trajectories 与 step-level CoT 一并保留，专家介入处暴露的纠错性操作本身成为高价值监督。TOUCAN [Xu25o] 在真实 MCP environments 中让多个强模型合成任务并执行完整 tool-use rollouts，经 task-level 与 trajectory-level 双重过滤形成大规模 tool-agentic corpus，多 teacher 的引入同时缓解了单一 teacher 带来的 distribution narrowing。ANCHOR [Wei26b] 从少量高质量 seed trajectory 识别 branch points，再由强模型围绕分支扩展新任务与新轨迹，seed 仅用于定位扩展位置，被外化的新轨迹仍由 teacher 参数生成。

**模拟环境合成（simulated-environment）。** 环境本身由模型扮演，$o$ 不再是真实返回，而是 simulator 的生成结果。ToolAlpaca [Tan23] 用 ChatGPT 分别扮演 user、assistant 与 tool executor，通过多智能体 simulation 生成 {Instruction, Actions, Response} 形式的工具使用轨迹，是较早清晰展示这一范式的工作。ToolMind [Yan25ab] 以 user / assistant / tool 三类 agent 的模拟生成带 reasoning 的多轮交互轨迹，并在 trajectory-level 与 turn-level 做细粒度过滤。ToolACE [Liu24o] 先自动扩充 API pool，再以多智能体 self-guided dialogue 合成单步、并行、依赖等多种复杂度的工具调用对话。APIGen-MT [Pra25b] 先生成 task blueprint 与 ground-truth action sequence，再经 simulated agent-human interplay 合成多轮 function-calling dialogues，以 rejection sampling 保留与 blueprint 一致的轨迹。Learning with Challenges [Kan26b] 先评估 student 的能力边界，再用多智能体生成器合成难度自适应的 mobile GUI demonstrations，使外化数据贴合 student 的 learning frontier。模拟环境下 $o$ 的真实性完全依赖 simulator 自身的保真度，verifier 的负担相应加重。

**构造式合成（constructed synthesis，弱交互）。** 此端几乎没有交互环境，trajectory 围绕预先给定的 answer、evidence 或问题结构反向构造，最接近纯参数读出。EviPath [Li25at] 从 gold evidence 反推 abductive reasoning path，再由强模型生成含 planning、sub-question decomposition 与 evidence-grounded answering 的 RAG trajectories。RAGShaper [Tao26] 构造带 distractor 的信息树与多跳问题，在受控检索环境中诱导强 teacher 产出含 thought、retrieval action、observation 与 final answer 的 agentic RAG trajectories。APIGen [Liu24n] 让强模型基于真实可执行 API 生成自然语言 query 与结构化 function-calling answers，经 format / execution / semantic 三层检查筛选——它保留了基于真实 API 的 execution check，但交互深度浅，整体偏生成而非交互，位置介于真实环境锚定与构造式合成之间。此端的外化轨迹一致性最高、最易规模化，代价是 $o$ 与真实环境的偏离风险也最大。

以上方法的目标载体都是 Narrative trajectory，但 Demonstration Externalization 的目标并不限于 Narrative。FABRIC [Ver25b] 通过 LLM-only pipeline 生成 task description、tool definition、policy pseudocode、dialogue 与 execution trace，其中 policy pseudocode 与 task / tool schema 属于 Schematic Tokenized，构成一个 **P7 → Schematic** 子格。此处归属须用前述判据确认：FABRIC 的输入是 prompt 而非已存在的 trajectory，被外化的程序骨架来自模型内化的能力，因而属 P7；若某工作是对一批已有 trajectory 做形式化抽取，则应归 P2 而非 P7。

Demonstration Externalization 的风险沿环境合成度轴分布。teacher 生成的轨迹可能包含 hallucinated actions、不可执行步骤或伪成功轨迹，这一风险在模拟环境与构造式合成两端显著放大——当 $o$ 由模型自身生成，一条“成功”的 observation 可能只是 teacher 的自我确认。单一 teacher 还会导致 distribution narrowing，使 student 过度拟合 teacher 的表达风格与决策偏好，multi-teacher mixing（如 TOUCAN）是常见缓解手段。

另有一类工作把 demonstration 外化与环境合成耦合在同一 pipeline 中。ASTRA [Tia26] 在合成 tool-use trajectories 的同时构造 reinforcement arenas，AgentSynth [Xie25e] 在合成 computer-use trajectories 的同时推进 scalable task generation 与 benchmark construction，COVERT [Xu26e] 在合成可验证 tool-use instances 的同时构造 RL training environment，Mock Worlds [Lu26j] 在合成 demonstrations 的同时造出 mock tool ecosystems。这些工作中 trajectory / instance 部分是合法的 P7，可在相应位置引用；但 reinforcement arena、mock environment、RL training environment 是未来经验的生成装置，不是经验载体，落在 §2.2 carrier 本体之外，不属 P7，按 §4 复合论文规则交由 §7 处理。把模型同时用作 UI simulator 与 teacher agent 的工作（如 Wan25as）同理，其 simulator 角色外化的是环境动态 / world model，亦不在 carrier 体系内。

### §6.2.2 Evaluative Supervision Externalization

Evaluative Supervision Externalization 指参数化 Evaluator——judge、verifier、critic、reward model——将其隐式评估能力外化为 Token-level supervision，供 student model、agent system、data filtering pipeline 或 reward-model training 消费。与 Demonstration Externalization 外化“如何行动”不同，这类方法外化的是“如何判断行动质量”。被外化的 source 仍是参数（Evaluator 的判断能力），被评估的 trajectory 只是判断所附着的 substrate，§6.2 的 P7/P1 判据在此依然成立，只是 source 换成了 Evaluator 参数。

目标 artifact 包括 critiques、step-level labels、action correctness annotations、progress labels、preference pairs、verification traces、failure diagnoses 等。按外化信号的粒度，这类方法可分为两支。

**步骤级正确性标注。** [He25g] 先用强 computer-use agent 生成 noisy trajectories，再由 o4-mini 对每一步做 correctness 与 optimality grading，并附 screenshot analysis、action review、alternative analysis 等 reasoning-rich annotations；评估结果被物化为 WebSTAR 与 WebSCORE 两套数据资源，前者训练 student computer-use agent，后者进一步蒸馏出轻量 StepRM。He25g 同时具有 Demonstration 侧贡献——WebSTAR 作为 student 训练数据——本节按其核心 artifact 归入 Evaluative，其 demonstration 侧已在 §6.2.1 述及。STEVE [Lu25e] 以 GPT-4o 作为 step verifier，比对动作执行前后的界面状态，为每一步生成 binary correctness label 与 verification reasoning，据此构建 step-verified trajectory data 并经 KTO 训练开源 student agent。

**长程进展与过程结构。** [Log26b] 用 judge 从 web task 抽取显式 constraints，在轨迹每一步评估 constraint satisfaction state，形成细粒度 CSR progress signal，并据此执行 prefix extraction 与 hindsight relabeling，使部分成功轨迹也能转化为可训练样本。Step-GUI [Yan25x] 先以轨迹级 verifier 建立质量锚点，再由强多模态 thinking model 把验证结果转写为 progress tracking、state summary、effect prediction、self-reflection、state verification、intent execution、action prediction 七类结构化 step-level supervision，把稀疏的轨迹级成败判断外化为密集的过程性监督。

Evaluative Supervision Externalization 需与 P6 区分。P6 的核心是 Evaluator 的 reward / preference / critique 直接参与 Policy 优化、使 Policy 参数更新；P7 强调 Evaluator 的输出被显式物化为 Tokenized artifact，且可脱离原 Evaluator 被其他系统复用。这给出了 §6.2.1 与 §6.2.2 的统一分界线：许多 Demonstration Externalization 方法用 judge 删除低质量轨迹，但若 judge 的输出只作为内部过滤开关，它就只是质量控制机制；只有当 Evaluator 的判断本身成为独立、可复用的 artifact，才归入 §6.2.2。这条线同样界定了本节与若干 evaluative-but-coupled 工作的边界——STeCa [Wan25x]、CGI [Yan25s]、ECHO [Li26l] 确实外化了 critique 或 calibrated supervision，但这些 artifact 主要服务于同一个同步演化的训练循环、未作为可迁移的数据资产被后续系统消费，更适合作为 P7 与 P6 / composite 之间的边界案例。

### §6.2.3 小结

Demonstration Externalization 与 Evaluative Supervision Externalization 分别外化 Policy 的决策能力与 Evaluator 的评估能力，对应“应该如何行动”与“如何判断行动质量”两条主线。一篇论文可能涉及多条转化路径，placement 取决于其主要被下游消费的 artifact 类型——这是一个可观察的机制事实（下游取走的是 trajectory 还是 label），与 §3.1 inclusion 判据“看机制不看自我定位”的精神一致。许多 agent data synthesis pipeline 同时含 teacher rollout 与 evaluator filtering，归类仍以核心 artifact 为准。

在框架层面，P7 在 §2.2.5 的 Experience Source 属性上占据一个特殊位置：{teacher}-sourced experience 几乎全部经由 P7 进入语料，P7 正是制造 {teacher} 这一经验来源的工艺过程。这也使 P7 区别于其余六条路径——P1–P6 处理的是已经存在的经验，P7 在生产新的经验载体。

P7 的核心 trade-off 由几个相互咬合的因素决定。能力天花板限定了 P7 不扩展 teacher 的能力前沿，它兑换的是成本、推理效率与领域专用化；相对人工标注，P7 以 teacher 的一次性推理成本换取近乎任意规模的监督数据，代价是数据质量受 teacher 能力与采样方差双重限制。verifier 在这条 trade-off 中扮演双重角色——既吸收采样方差，又通过 best-of-N 效应把 filtered student 的可靠性抬到 teacher 平均水平之上；环境合成度则决定 $o$ 的 grounding 强度与 verifier 的负担轻重。environment-mediated synthesis、multi-teacher mixing、curriculum 与 difficulty-aware sampling、execution-based verification、step-level filtering、human-in-the-loop screening 这些横向设计维度，作用对象都是外化 artifact 的质量、规模、多样性与可验证性这几个量。
