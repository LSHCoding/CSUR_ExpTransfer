# §3.1 Narrative-to-Narrative Transformation（P1）

P1 将 Narrative Tokenized 的 Agent Experience 转化为更高层的 Narrative artifact。源经验是 Agent 在具体上下文下观察、推理、行动并接收环境反馈所产生的自然语言或多模态序列化记录。P1 不改变经验所处的载体层级——源端与目标端都位于 Tokenized 层内的弱形式化区间——它在该层内完成抽象、压缩、重写或归纳。转化后的 artifact 形式多样，包括 reflection、summary、lesson、rule、guideline、insight、causal statement 与 natural-language skill，共同点是后续 Agent 通过 language / multimodal understanding 直接阅读这些 artifact，经 retrieval-augmented prompting 或 prompt injection 进入决策上下文，而非交由 parser、executor、graph traversal 或训练过程处理。

P1 文献在命名同类对象时高度不一致。skill 一词在不同工作中分别指代可执行代码、自然语言步骤说明与分层知识库条目；memory、experience、reflection 同样既同名异物又同物异名。本节的归类不采用论文自报的术语，而依据两个可观测属性：artifact 的产物形态（自然语言文本、可解析可执行的结构、还是图），以及下游复用机制（由 LLM 阅读理解，还是由 executor / parser / graph traversal 处理）。论文原始术语仅在介绍具体工作时保留。按此口径，产物为弱形式化文本、复用靠阅读的工作归入 P1；产物具备会被机制性利用的结构（code、workflow、graph）的工作归入 P2（§3.2）；artifact 不来自具体轨迹、而由模型凭参数知识直接生成的工作归入 P7（§6）。这条判定原则同样适用于 §3.2，宜在 §3 章首统一声明。

P1 工作之间的实质差异不在 artifact 的表面名称，而在经验库的维护方式。本节先按经验库类型二分。Static Memory Store 指经验库一次性或批量构建后、对已有条目只读复用；纯追加扩充、入库前对原始轨迹做过滤、仅在任务内维护临时缓存、乃至完全不维护持久库，都属此类。Dynamic Memory Store 指经验库在 Agent 持续运行过程中对已有条目执行改写、合并、淘汰、替换或版本更新。两类的判别点是已有条目是否会被回改，而不是是否有新条目写入——只增不改的库仍属 Static。Static 库内部再按经验抽象所依赖的轨迹数量分为单轨迹与多轨迹两类。Dynamic 库的写入呈流式到达，单次写入通常仍以单条轨迹为单位，其特征在于跨时间的条目演化，单条与多条的对立在此不再成立，故不再细分。

P1 与 P7 的边界由源端决定：P1 的输入必须是具体 Agent trajectory 或 interaction log，若 artifact 主要由模型依据任务描述或内化知识直接生成、无具体轨迹作为输入，则属 P7。

## §3.1.1 Static Memory Store（静态经验库）

Static Memory Store 类方法把经验抽象的产物当作只读资产积累。经验库的具体形态不一：有的是任务内临时缓存，任务结束即弃；有的是构建后不再回改的持久库；有的在抽象前先对原始轨迹做过滤。它们的共性是条目一旦写入便不再被编辑或合并，经验质量由单次抽象的好坏决定，系统在长期使用中没有自我纠错的环节。这类设计实现简单、行为可预测，代价是错误一旦进入库便会持续影响后续决策。

下面按抽象所依赖的轨迹数量分两类讨论。

### §3.1.1.1 Single-Trajectory Experience Abstraction（单轨迹经验抽象）

单轨迹经验抽象以一条 Agent trajectory、episode 或局部 execution log 为输入，把一次具体执行压缩为更高层的自然语言 artifact。触发时机通常是任务结束、失败发生或一次 trial 完成，手段包括 reflection、summarization、critique、failure explanation。单条轨迹本身已携带可复用信号：失败轨迹可能暴露错误假设、无效动作或遗漏约束，成功轨迹可能包含有效的分解策略或环境交互模式。这类方法反应快、成本低、不依赖历史数据积累，适合 trial-and-error、online adaptation 与任务分布较窄的场景；局限在于抽象依据单条轨迹，容易把偶然因素误写为通用经验，且与原任务上下文绑定较强，跨任务迁移能力有限。

Reflexion [Shi23b] 是这一范式的早期代表。Agent 在每次 trial 结束后，依据失败轨迹与环境反馈生成简短 reflection，将错误归因与改进建议写成 verbal memory 拼接进下一轮 prompt；消融实验显示，事后生成的 reflection 比直接回放原始 episode 更有效。Reflexion 的经验库是典型的任务内缓存——旧条目本身不被改写，仅在上下文超长时按时间截断。MemOrb [Hua25] 在客服场景下把每轮对话及其工具调用轨迹事后压缩为一个 Orb，以六元组形式记录 observation、emotion、outcome、context 与策略性反思，存入可检索层后由后续对话按需取回；其 rewrite 环节针对的是查询而非已有 Orb 条目。Reflection-Based Memory [Aza25] 面向 web navigation，将单条浏览轨迹总结为 Web-Reflection，内容覆盖有效步骤、站点功能局限、捷径与失败回溯点，以任务 embedding 为键存入 key-value 知识库，测试时检索 top-k 注入 prompt。

REFLECT [Liu23] 处理机器人单次执行的多模态观测流，先把 RGB-D 视频、音频与状态记录转为 scene graph、事件级 caption 与 subgoal-level summary，再做分层摘要，并在子目标验证失败时生成 failure explanation 驱动 correction planning。该工作不维护跨 episode 的持久库，整个流程是一次性的单轨迹总结。WebCoach [Liu25e] 把每个已完成的 web episode 压缩为 3 到 5 句 summary，保留 success status 与 fail mode 等字段后存入 External Memory Store；它只持久化已完成 episode、丢弃半成品过程，这种入库前过滤不构成对库内条目的演化。

代码域的两项工作把单轨迹抽象推向字段化的经验卡片。SWE-Exp [Che25d] 从单次 repair trajectory 中分别提炼 comprehension experience 与 modification experience，形成带 perspective、modification、issue type 的 experience bank 条目；论文强调 bank 的持续扩充，但未给出对旧条目的编辑或合并机制，按只增不改的口径仍属 Static。MemGovern [Wan26ap] 从 GitHub issue、PR、patch 的闭环修复记录中提纯 Experience Card，经 checklist 式 refine loop 保证字段质量，产物分 Index 层与 Resolution 层。MemGovern 的经验源是 governed human experience 而非 Agent 自身试错——这与本节多数工作不同，但 Project_Infos 的纳入标准对经验来源不设限，人类示范同样合法。

Meta-Policy Reflexion [Wu25] 把失败 episode 的 retrospective reflection 写成带权重的 predicate-like 规则，而非长文本反思段落；新规则在训练回合中并入 memory，测试阶段 memory 冻结，Agent 检索规则注入 prompt 并做 admissibility check。规则在测试期不再变动，这使它落在 Static 一侧；其规则形式比一般 natural-language reflection 更形式化，接近轻量 rule set，但复用仍靠阅读而非执行，归 P1。Procedural Knowledge at Scale [Wu26] 把 teacher model 合成的大规模 reasoning traces 切成 subquestion-subroutine 对，构成静态 procedural datastore，推理时模型先 verbalize 当前子问题再检索对应 subroutine 作为 hint。其经验源是 teacher-synthesized traces、实验形态接近 reasoning 而非 Agent-environment 交互，可作为本节的外延材料。

### §3.1.1.2 Multi-Trajectory Experience Induction（多轨迹经验归纳）

多轨迹经验归纳以多条 trajectories、demonstrations 或 failure cases 为输入，通过比较、聚类或合并抽取更稳定、更可迁移的 Narrative artifact。可迁移经验通常来自多次执行之间反复出现的稳定结构，对成功与失败轨迹做对照可以识别哪些行为与任务完成更相关、哪些错误跨上下文反复出现。相比单轨迹抽象，这类方法更可能滤掉单次轨迹的偶然噪声，代价是冷启动能力弱、且存在过度归纳的风险。

本小节在 Static 范畴内可归入的工作较少。多数做跨轨迹归纳的方法同时维护一个会演化的经验库，按本节口径落入 §3.1.2；只做一次性跨轨迹归纳、之后只读复用的工作构成 Static 多轨迹这一较小的集合。

AutoGuide [Fu24] 从同一任务的优劣轨迹对中定位首次决策分叉点，围绕分叉动作抽出与上下文绑定的 guideline，写成 when-context then-action 形式并按 context 建成字典；单个 guideline 依赖一对成功与失败轨迹的对照，是典型的 pairwise contrastive extraction。Coarse-to-Fine Grounded Memory [Yan25] 先从环境基础信息与离线 trial-and-error 轨迹得到 coarse focus points，再比较成功与失败轨迹提炼 hybrid-grained tips，其长期可复用部分主要是 tips dictionary；在线阶段抽取的 fine-grained key information 更像当前轨迹内的临时分析，不进入长期库。

M2 [Yan26b] 把单轨迹压缩与多轨迹归纳放在同一系统的两层记忆里。internal memory 在每步把 observation、thought、action 压成简短状态摘要，递归拼接成紧凑历史；external insight bank 则用 strategy abstractor 从历史成功日志中提炼 topic-tagged insights。两层库都不对已有条目做回改，均属 Static——M2 因此横跨单轨迹与多轨迹两类，internal memory 对应单轨迹压缩，insight bank 对应跨日志归纳。

## §3.1.2 Dynamic Memory Store（动态经验库）

Dynamic Memory Store 类方法把经验库当作随交互持续演化的对象。系统不仅写入新经验，还对已有条目做改写、合并、淘汰、替换或版本更新，使经验库在长期使用中具备自我纠错与去冗的能力。这类设计的核心问题不再是“如何从一次经验生成一个 artifact”，而是经验如何进入长期记忆、已有记忆如何被更新、重复或过时条目如何被合并或删除。动态库的写入是流式的，单次写入往往仍是单条轨迹，其区别性特征在跨时间的条目演化，本节因此不再按单条/多条细分，而按演化机制的差异组织叙述。

一类工作维护自然语言规则或 insight 库，并把编辑与投票写成显式 memory operation。ExpeL [Zha23c] 用专门的 insight model 从成功轨迹池与失败-成功对中抽取可迁移 insight，对已有条目执行 ADD、EDIT、UPVOTE、DOWNVOTE，每条 insight 带重要度并可随后续经验被修改或移除。H2R [Ye25b] 把 hindsight reflection 拆成高层 planning memory 与低层 execution memory 两套，memory pool 同样支持 add、modify、upvote、downvote，planner 与 executor 各自检索对应层。AutoManual [Che24] 让 Planner 在环境中交互，Builder 按 direct success、indirect success、failure 三类结果差异化地抽取规则，Consolidator 在规则过多时合并冗余、删除过时项，Formulator 再整理成 Markdown manual——这是边写入边巩固的两阶段演化；旧版分类曾把 AutoManual 与 ExpeL 放在跨轨迹归纳一节，按经验库是否演化的口径，二者同属动态。Agentic Context Engineering [Zha25f] 由 Reflector 从 reasoning trajectory 抽出 concrete insight、failure mode 与 strategy bullet，Curator 把它们作为 delta 合并进 Context Playbook，每条 bullet 带 id 与 helpful / harmful 计数，并配套语义去重与低效条目裁剪。

另一类工作以替换、淘汰或 survival 评分驱动经验库收敛。Darwinian Memory [Mi26b] 把每段 GUI 经验写成以 precondition 与 goal 为索引的 memory unit，通过 evolutionary replacement、survival-based pruning 与 reputation-style inhibition 调节记忆库，后续执行命中 memory 后直接复用其轨迹，必要时以 epsilon-mutation 探索更优版本。R2D2 [Hua25e] 把 web 轨迹截断、纠错并附加 reflective insight 形成 query-keyed 记忆，当新轨迹能为同类 query 提供更准确反思时更新旧值。Remember Me, Refine Me [Cao25] 把经验写成带 usage scenario、confidence、tools 的结构化条目，持续按 utility 做 pruning——其动态性主要来自删除与筛选，而非对旧条目正文的逐字重写。Mistake Notebook Learning [Su25] 以一批失败样本为更新单元，先按 subject 聚类、再生成 mistake note，每轮更新都要经整批 post-update evaluation，只有新版 notebook 整体提升时才接受；它的更新是批量而非在线逐条，接受与回滚机制比常见 append-only notebook 更严格，正是这种受控的版本替换使它属于动态。Hindsight Trajectory Rewriting [Hu25] 从单条失败轨迹经 hindsight rewriting 重写出针对可达替代目标的合成成功 workflow，写入 replay buffer，当同一目标出现更短更优版本时 buffer 会替换旧条目。

围绕 skill 的工作把单次经验不断吸收进持续演化的技能库。Trace2Skill [Ni26] 由并行 analyst 对各条成功与失败轨迹提出局部 patch，再经分层 merge 与 conflict-free consolidation 合成可迁移 skill 更新，产物是以 SKILL.md 为核心的技能文档，已有文档可在 deepening mode 下被进一步改写。AutoSkill [Yan26] 从用户交互信号中抽取 skill candidate，经 prompt-driven judge 决定 add、merge 或 discard，对保留技能做版本化演化，产物是含 trigger、tags、examples 与 version 字段的 SKILL.md。SkillClaw [Ma26b] 汇总多用户、多 agent 的真实使用 session，由 Agentic Evolver 判断 create、refine 还是 skip，夜间验证后把有效更新同步回共享 SkillHub。这三者的承载物虽是带字段的技能文件，但复用时由 Agent 直接阅读 SKILL.md 正文、不经专门执行器或解析器，按下游消费机制归 P1。

另一些技能或过程库的结构化、多模态程度更高，处在 P1 与 P2 的边界。SkillX [Wan26] 构建 planning、functional、atomic 三层 SkillKB，经 merge、filter 与 exploratory expansion 持续优化；其 planning 层可写成自然语言步骤，functional 与 atomic 层的结构化程度明显更强。Skill Set Optimization [Not24] 从 actor 历史中抽取高价值 subtrajectory，概括成带 subgoal 与编号指令的 skill，skill set 依实际 observed value 被持续筛除与替换。Memp [Fan25] 同时容纳 verbatim trajectory 与 abstract script 两种形态的 procedural memory，在成功或失败后执行 add、adjust、delete，并可把新旧信息合成 revised memory。GUI-explorer [Xie25] 为每次状态转移抽出 visual key 与对应 textual operational value，当 key 匹配而 value 足够不同时把新值并入旧条目，产物是带视觉键的多模态条目。Mirage-1 [Xie25b] 把单条 GUI 轨迹概括成 execution skill，再上卷为 core skill 与 meta skill，新轨迹到来时由 RefineSkillModule 增补合并。Get Experience from Practice [Fen25] 把多轮 interaction trace 总结为高层 procedural experience、低层执行 experience 与配套 check function，在 repository 中做 updating、merging 与 ranking——其低层项可直接转为 API 调用或脚本，这部分明显带可执行性，与纯 narrative artifact 有距离，接近 P2，落稿前需对其主归属单独裁定。

还有一类工作并行维护多个语义分工不同的演化库。Learning on the Job [Yan25d] 把长程交互分别转成 strategic memory 的 dilemma-strategy 对、procedural memory 的 SOP 与 tool memory 的操作提示，任务后统一做 refinement、integration 与 deduplication。What Deserves Memory [Ma25] 用原始 episode 与 anticipatory schema 的 prediction error 判定哪些信息值得保留，蒸馏成 semantic knowledge，并把连续 episode 融合成 narrative episodic memory，两层库都执行 merge 与 conflict replacement。CLIN [Maj23] 从每次 trial 抽取因果式语言陈述（如某动作对某结果 necessary 或 does not contribute），并由优质 memory 再总结出跨 episode 的 meta-memory，每次 trial 后都生成更新版本。Contextual Experience Replay [Liu25] 把每次经验蒸馏成 environment dynamics 与 skills 两类摘要，写入 buffer 时显式参考已有内容以避免重复，当前写入因此受历史整合状态约束。

Dynamic Cheatsheet [Suz25] 的更新对象是整份文档而非单个条目。Curator 对测试时积累的解题轨迹做 usefulness 与 generalizability 评估，把可迁移内容蒸馏进 cheatsheet，并对其中条目执行改写、删除、合并与压缩以控制长度，其更新粒度比常见的条目式 memory 更粗。

## §3.1.3 讨论

Static 与 Dynamic 的取舍对应一组明确的 trade-off。Static 库实现简单、行为可预测、运行期开销低，但经验质量在写入时即被锁定，系统缺少自我纠错环节，一条被误写的 reflection 会持续影响后续检索。Dynamic 库通过 merge、replacement、pruning 在长期使用中收敛，更能滤除单次抽象的偶然噪声，代价是引入了冲突消解、staleness 判定、版本管理等额外机制，行为更难复现，且淘汰策略本身可能误删仍然有用的条目。

在 Static 内部，单轨迹与多轨迹的对立同样是一组 trade-off。单轨迹抽象延迟低、数据需求小、冷启动表现好，适合任务分布窄、需要快速适应的场景，但单条轨迹的偶然性与错误归因会直接进入经验库。多轨迹归纳通过跨轨迹对照获得更强的泛化性与抗噪性，代价是冷启动能力弱，且不同轨迹可能支持相互冲突的结论，需要额外的适用条件建模。Static 多轨迹一类工作数量偏少，本身是一个值得记录的观察：跨轨迹归纳一旦展开，往往伴随经验库的持续演化，纯静态的跨轨迹归纳是较窄的设计空间。

P1 工作的经验来源以 Agent 自生成轨迹为主，MemGovern 取自人类修复记录、Wu26 取自 teacher 合成 traces 属于例外；经验来源与 Static / Dynamic 的划分正交，不改变 pathway 归属。多模态以正交属性形式进入 P1：REFLECT 处理机器人多模态观测流，GUI-explorer 与 Mirage-1 的技能条目带视觉键，这些工作并未因模态不同而走单独路径，只在各自类别内呈现实现差异。

P1 内部从 Static 单轨迹、Static 多轨迹到 Dynamic，经验库从一次性产物走向持续演化的资产。哪一种更合适并无普适答案，取决于下游任务对 cold-start、泛化性、运行开销与可复现性的约束权重，§8 将把这一选择与其他 pathway 放在同一框架下比较。
