

## §5.2 Tokenized-to-Policy Transformation（P5）

P5 把离散化的 Agent Experience 内化为 Policy/Actor 参数。源经验是各类 Tokenized artifacts——自然语言轨迹、工具调用记录、代码编辑历史、GUI 操作序列、reasoning chains;转化的目标是让这些经验进入 Policy weights,使模型在后续任务中无需显式检索、拼接或重放原始经验即可完成序贯决策。

P5 要回答的不是如何训练一个更强的模型,而是 Agent 在交互中形成的离散经验如何被组织成训练信号、并改写 Policy 的参数化决策能力。与目标载体为可读文本、规则(P1)或程序、工作流(P2)不同,P5 的目标载体是不可读、不可局部寻址的权重;经验内化后调用成本低、不受 context window 限制,代价是显式可追踪性与局部可修改性下降。

本文按一条轴把 P5 分为三类:policy update 消费了一条轨迹经验的哪个侧面。Imitation-based Policy Internalization 消费成功轨迹的动作序列本身,让模型直接模仿;Environment-Reward-based Policy Internalization 消费每条轨迹的客观结果,用可验证的环境反馈优化;Preference-based Policy Internalization 消费轨迹之间的相对序,构造偏好并优化。

P5 与 P6 的边界由 reward 或 preference 信号的来源决定。信号是非参数化的——环境可验证结果、unit test、ground-truth match、rule-based verifier——经验直接走 Tokenized → Policy,属 P5。信号来自参数化 evaluator(trained RM、PRM、LLM-as-a-judge)时,转化先经 Tokenized → Evaluator(P4)、再 Evaluator → Policy(P6),是复合路径,不在本节。参数化 evaluator 若仅用于筛选或重构训练数据、其分数不进入 policy update 的 loss,经验仍是直接内化,工作仍归 P5。

### §5.2.1 Imitation-based Policy Internalization

Imitation-based Policy Internalization 把高质量 Agent trajectories 当作监督示范,通过 behavior cloning、SFT 或 rejection-sampled fine-tuning 训练 Policy。它消费的是成功轨迹的动作序列本身:若一组轨迹已体现有效的任务解决行为,模型经模仿其动作选择、推理模式、工具调用习惯或 GUI 操作序列即可把相应经验内化进参数。环境反馈在这里不进入优化目标,只承担 data selection 与 data construction——筛掉失败 rollout、过滤无效工具调用、保留通过测试的代码修改。

主线机制是 success-filtered behavior cloning,环境信号决定哪些数据进训练集、loss 始终是成功数据上的最大似然。一项 web agent 工作以 environment error 与 self-critique 过滤 WebArena 中的自生成轨迹,在 QLoRA 下做 autoregressive SFT [Pat24];SWE-Gym 用可执行 unit test 判定 patch trajectory 成败,以 rejection-sampling fine-tuning 完成 policy update,同文另训的 verifier 只做 reranking、不改变 agent 的归类 [Pan24];Go-Browse 通过 structured exploration 采集 web trajectories,仅对成功轨做 SFT,成功标签由 VLM-as-a-judge 给出但仅用于筛选 [Gan25b]。

一类工作在 imitation 前对轨迹做轻量加工,以提升示范信号的质量与覆盖。NNetNav 对开放网页中的无监督探索轨迹做 hindsight instruction relabeling,把无目的交互改写成 instruction-following demonstrations 后 SFT [Mur24b];AndroidGen 用 StepCritic 把长 Android 轨迹切成成功子段并做 augmentation,再在筛后轨迹上做 LoRA-SFT [Lai25d]。这类加工只重排或重标注轨迹内容、不改变其载体层次,仍是 P5 内部的预处理。若加工把经验跨载体重构,则构成与 P1 或 P2 的链式复合,归 §7 处理:WebCoT 把 reflection、branching、rollback 痕迹 verbalize 成显式 CoT 再内化 [Hu25i],AgentTrek 把网页 tutorial 编译成 Goal-Observation-Reasoning-Action schema 再 SFT [Xu24]。

数篇工作冠以 RFT、ReST 或 RL 之名,最终的 policy update 仍是 success-filtered 数据上的最大似然。AppVLM 称 ReST 或 RFT,但失败轨 return 为零、不进训练集,核心更新是 success-only SFT [Pap25b];一项 VLA 工作的两阶段框架中,Stage 1 用 binary reward 在 action head 上做探索、Stage 2 对全模型做 MSE supervised learning,reward 的作用是 data discovery、full-model 的实际更新机制是 imitation [Guo25d]。判定 P5 子类依据的是 policy weight 被改写时所用的 objective,而非论文的方法命名。

Imitation-based 适合 Agent 的 cold-start,能快速教会模型动作空间、输出协议与基本交互流程,AgentTuning 进一步把 agent trajectories 与 general instruction data 混训以保住通用能力 [Zen23d]。能力上限受示范覆盖限制——训练数据只含有限场景的成功路径时,模型易学到表层格式或局部启发式,难以主动探索新策略。经验写进权重后即不可局部寻址,新增或修正一类行为需重新 fine-tune,这一可编辑性代价与 P5 另两类共通,跨 pathway 的取舍见 §8。

### §5.2.2 Environment-Reward-based Policy Internalization

Environment-Reward-based Policy Internalization 让 Agent 与环境交互产生 trajectories,用非参数化、可验证的反馈信号直接优化 Policy。它消费的是每条轨迹的客观结果:任务成败、unit test pass/fail、代码执行反馈、SQL execution correctness、ground-truth answer match、GUI 任务完成状态、机器人操作成败——这些均由环境本身可程序判定。命名中的 Environment-Reward 用以划界:reward 来自 trained RM 或 LLM-as-a-judge 的工作归 P6。与 Imitation 只学成功轨迹做了什么不同,这里失败轨迹经 reward 转成负信号,同样进入优化。

基本前提由早期工作确立。GLAM 把 FLAN-T5 放入 BabyAI-Text,以任务成功 sparse reward 经 PPO 做 online grounding [Car23];LLaRP 连接冻结的 LLaMA 与视觉编码器为 embodied policy,以 success、subgoal completion、invalid-action penalty 组成的环境 reward 经 DD-PPO 训练 adapter 与 heads [Szo23]。只要 Agent 动作可在环境中执行、环境能返回客观可验证的结果,Tokenized trajectory 即可直接转化为 Policy weights;在这些短程或单目标设定下,稀疏 reward 配合 PPO 已足够。

当设定推进到长程、多轮的真实环境,这个前提开始承压。WebAgent-R1 以 M-GRPO 并行采样 browser trajectories、reward 由 URL match 与 Playwright verification 给出 [Wei25],一项长上下文 SWE 工作用 DAPO、reward 来自 validation test 通过与否加 length penalty [Gol25];它们把 RL 推进到真实多轮环境,同时也暴露出核心困难——一条几十步的轨迹只在终局拿到一个稀疏的可验证 reward,这个信号很难被有效利用。让一个稀疏而可验证的 reward 在长程多轮下变得可用,是此后工作的主线问题;它沿三层推进——让信号不消失、让信号可归因、让信号在源头变密。

第一层是让信号不消失。长程 Agent 在训练早期极易坍缩到低探索策略,一旦坍缩,policy 不再产生成功轨迹,稀疏 reward 下便没有正样本可学。一组方法把 entropy 作为可调控量来对抗坍缩:EPO 在 PPO 或 GRPO 外层加 trajectory-level entropy regularization 与 entropy smoothing,缓解 sparse-reward multi-turn RL 的 exploration-exploitation cascade failure [Xu25k];Agentic Reinforced Policy Optimization 在 multi-turn tool-use 中用 entropy-based adaptive rollout,在高不确定性步骤触发局部分支采样,并对 shared 前缀与 branched 动作分配不同 advantage [Don25d];AEPO 做 entropy-balanced clipping 与 rollout budget 动态分配 [Don25c],EMPG 按当前 entropy 缩放每步 advantage、并加 future-clarity bonus 鼓励进入低熵后继状态 [Wan25ad]。另一组方法在信号已经稀缺时直接补正样本:一项 GUI 工作维护成功 trajectory replay buffer,当采样组全失败、无梯度时从历史缓冲注回正样本 [Lu25f];SoLS 做 off-policy 的非对称更新,正优势样本强更新、负优势样本保守正则,并配合成功转移回放 [Pap25c];AgentGym-RL 的 ScalingInter-RL 用先短后长的 horizon curriculum,让 Agent 先在可完成的短程任务上拿到信号、再逐步延长 [Xi25c];UI-S1 走 semi-online,rollout 用模型自生成历史、在偏离 expert 时由 patch module 续接 [Lu25j]。

信号能持续产生之后,第二层是让它可归因。多轮轨迹里 reward 常只在终局给出,而决定成败的可能是中间某一步;把一个稀疏 return 准确分配到决策单元,是这一层的核心。一条清晰的演化线是归因粒度的持续下沉:从整轨,到 turn-level,到 step-level。ArCHer 用环境 reward 先训 utterance-level Q critic、再把 multi-turn advantage 传给 token actor [Zho24f];Turn-PPO 把多轮交互建模为 turn-level MDP,在 turn 粒度上用 critic 做 GAE [Li25ae];GiGPO 用 anchor-state grouping 比较不同 rollout 在相同中间状态下的动作,同时构造 episode-level 与 step-level relative advantage [Fen25c];IGPO 用每轮交互前后 ground-truth answer log-prob 的增量定义 information-gain intrinsic reward,把 observation 的真实信息贡献转成 turn-level 信号 [Wan25y]。与粒度下沉并行的一支取向是回避 learned critic 的方差与开销:GiGPO 的 group-relative 估计、LOOP 的 leave-one-out PPO 都用 value-free 方式替代显式 value network [Fen25c, Che25af],Q-SFT 则把 offline 数据的 Bellman target 当作 token cross-entropy 的 sample weight,形式像 SFT、机制是 value-based credit assignment [Hon24]。actor-critic 中的 critic 属 V-Par,这会引出一个归类疑问:它是否已落入 P6?critic 由环境 reward 在 RL 算法内部 bootstrap 而来,是估计 expected return 的内部辅助件,并非独立训练的外部 reward 来源;reward 的 provenance 仍是环境,ArCHer、Turn-PPO 因而仍属 P5,与 P6 中 evaluator 充当 reward 来源有本质区别。

第三层比前两层更靠近根部:与其在一个固定的稀疏 reward 上更努力地探索或归因,不如让 reward 信号本身在源头变密。做法是从环境里挖掘可程序验证的中间结构,把单一终局 flag 重写成 composite reward。ToolRL 把 reward 拆成 format correctness 与 tool name、arg name、arg value 的三层匹配 [Qia25b];SQL-Trail 在 reason-execute-observe 循环里用 execution、turn-budget、schema grounding、syntax、format 等六项组成 composite reward [Hua26d];SQL-ASTRA 的 Column-Set Matching Reward 从结果表列值集合的比较中给出 dense partial-credit [Li26r];RLVMR 对 planning、explore、reflection、monitor 等 meta-reasoning 行为给 rule-verifiable process reward [Zha25ao]。在 GUI 与 VLA 域,rule-based verifiable reward 被直接用作 GRPO 的输入——format 合法性叠加 action-type、click-point、text-match 等可判定项 [Luo25b, Zha25an, Li25aa]。环境越复杂,高价值的 reward 越不在终局那个 flag,而在那些可程序验证的中间结构,被逐步挖掘并重写进 optimization objective。credit assignment 把一个固定的稀疏信号归因得更准,reward construction 让信号在源头就更密;近年的重心正从前者向后者移动。

另有一支工作聚焦把这套训练规模化:AgentRL 以 fully asynchronous rollout 与 task advantage normalization 支撑多任务混训 [Zha25ag],[Gu26b] 在大规模 mobile emulator 上做异步在线 RL、并以 instance、template、app 三层考察泛化;这些关切偏向系统与 benchmark,与信号可用性正交。

Environment-Reward-based 的能力上限高于 Imitation——Policy 能通过试错发现超出示范数据的新策略——代价是 reward 稀疏、credit assignment 困难、online rollout 昂贵以及 reward hacking。它与 Imitation 共享经验写进权重后不可局部寻址的性质,并多一层不可逆:RL 更新由高方差梯度驱动,一次坍缩或 reward hacking 造成的能力损伤,难以像编辑一条文本规则那样回退。实践中 imitation 与 reward 常是同一 pipeline 的两个阶段——cold-start 用 imitation 立起格式,能力突破交给 reward optimization;前述 SWE 工作先用通过测试的轨迹做 RFT 热启动再上 DAPO [Gol25],SQL-Trail 与 AgentCPM-GUI 也都先 SFT warm-up 再 RL [Hua26d, Zha25an]。

### §5.2.3 Preference-based Policy Internalization

Preference-based Policy Internalization 从 Agent experience 中构造轨迹间的相对偏好,通过 DPO、DMPO 或其他 trajectory-pair optimization 更新 Policy。它消费的是轨迹之间的相对序:相对更优的轨迹来自成功任务、通过验证的执行、修正后的轨迹或更高效的操作路径,相对较差的来自失败尝试、错误工具调用、未通过测试的代码修改。动机直接——许多 Agent 场景中精确设计每步 reward 很难,但判断两条轨迹孰优孰劣相对容易。

相对另两类,Preference-based 显式消费失败经验:Imitation 丢弃失败轨,Environment-Reward 把失败轨压成一个负 scalar,Preference 则把失败轨作为 negative、与成功或修正轨配对,让 Policy 学到应回避的行为。DMPO 把 DPO 从单轮 response preference 扩到 multi-turn trajectory preference,引入 state-action occupancy measure 与 length normalization 解决长轨迹下的 preference objective [Shi24c];ETO 先用 expert trajectory 做 behavior cloning,再让 Policy 主动探索,把成功示范与探索中的失败轨组成 preference pairs 用 DPO 持续更新 [Son24];Agent-RLVR 针对 SWE agent 奖励稀疏,用 guidance 帮助模型探索更可能成功的修复轨迹,再依 unit test pass/fail 构造 preference pairs——其核心 objective 是 DPO、偏好标签来自非参数化的 unit test,故归 Preference-based [Da25];一项 tool-use 工作从 inference tree 中挖掘失败分支,把成功路径关键步与同层失败兄弟节点配对成 step-level preference [Che24k]。

偏好粒度同样在下沉,与 §5.2.2 中 credit assignment 的演化平行。HPL 指出长程任务只做 trajectory-level comparison 过粗,联合 trajectory、group、step 三层偏好做分层优化,group 由 fixed、entropy 或 semantic segmentation 划出 [Gao25c];WEPO 把偏好收缩到 web element 层面,用目标元素对干扰元素的对比训练提升局部动作判别 [Liu25z]。Environment-Reward 与 Preference 两条路线各自独立地走向更细的归因或对比粒度。

与 Environment-Reward 相比,Preference-based 是 supervised-style 训练,不需直接估计高方差 policy gradient,训练更稳;与 Imitation 相比,它能用负样本与轨迹对比,更适合处理失败、修正与探索经验。局限在于 preference pair 的构造质量直接决定效果,trajectory-level preference 仍可能缺乏细粒度 credit assignment。它与另两类同属把经验写进权重的 P5、内化后同样不可局部寻址;相对 Environment-Reward,其训练过程更接近 SFT、无高方差 RL 梯度带来的剧烈坍缩,代价是表达力受限于成对比较——一条轨迹"好多少"无法编码,只有"更好"。

三类的判定标准对应 §5.2 的划分轴。核心 objective 是 SFT、behavior cloning 或 imitation,environment feedback 仅用于筛选或构造样本,归 Imitation-based;核心训练信号是非参数化的环境反馈、执行反馈、规则化 verifier 或 ground truth,且直接进入 policy optimization,归 Environment-Reward-based;核心 objective 是 preference 或 trajectory-pair optimization,且偏好标签来自非参数化信号,归 Preference-based。reward 或 preference 若主要来自 trained reward model、LLM-as-a-judge 等参数化 evaluator,则属 Evaluator-mediated Policy Optimization,归 P6。
