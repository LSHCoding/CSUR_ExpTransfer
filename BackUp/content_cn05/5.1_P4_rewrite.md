## §5.1 Tokenized-to-Evaluator Transformation（P4）

P4 把 Tokenized agent experience 内化为参数化 Evaluator。源经验包括 agent trajectory、reasoning trace、tool-use log、action history、preference pair、execution outcome、failure annotation 等离散记录；经过训练，其中的评价语义被写入 Evaluator 参数，形成可复用的评估能力。训练后的 Evaluator 表现为 outcome reward model、process reward model、verifier、critic、judge、value model、failure detector 或 diagnostic feedback model。

判定一项工作是否属于 P4 的硬标准是 **Evaluator 参数是否被经验数据实质性更新**。只在推理时通过 prompt 激发已有模型评估能力、参数不发生更新的 prompted LLM-as-a-judge 不在此列。判定只看参数是否被更新，不限训练手段——SFT、RL、self-training、对比学习均可作为内化方式，本节内部确实并存这几类方法。

本节沿两条相互独立的轴组织 P4 工作。

**主轴是监督粒度（supervision granularity）**，区分训练时监督信号被分配到的层次。**Outcome-supervised** 把信号分配在完整输出、完整轨迹或完整 episode 上——任务是否完成、最终答案是否正确、两个完整 response 哪个更优。**Process-supervised** 把信号分配在中间步骤、动作前缀或单个 decision 上——某个 reasoning step、tool call、GUI action、state-action 对是否正确、是否有进展。粒度判定看监督信号挂在何处，而非任务本身有多少步：一个多步 agent 任务若 Evaluator 只在终局成败拿到 label，仍归 outcome。两个子类互斥，构成 §5.1.1 与 §5.1.2。

**次轴是产物形态**，区分训练后 Evaluator 推理时输出什么。**判别式（discriminative）评估器**的产物只有标量或可直接读为标量的信号——单个分数、binary 正确性、pairwise 偏好概率、reward 向量。**生成式（generative）评估器**的产物包含自然语言或结构化生成内容——critique、error diagnosis、failure explanation、verification rationale、repair suggestion、structured justification。这条次轴不取「scalar / non-scalar」的字面二分：相当一批工作的产物兼有标量与生成内容（先生成 rationale 再给 verdict），字面二分会让它们无处归属。判别式与生成式之间存在单向的信息投影关系——一个生成式评估器的产物总能塌缩出标量（verdict token 读作概率即可），反向则不成立，标量推不出 critique。一项工作只要其方法贡献包含生成式产物，就按其更富的成分归入生成式子类。次轴在 §5.1.1 与 §5.1.2 内部各切一刀。

少数 Evaluator 同时携带 outcome 与 process 两种监督。这类工作按其**主导监督信号**归入对应子节，并在文中标注次级成分；不另设独立类别——outcome 与 process 在这些工作中是程度差异而非类别对立，强行并列会破坏主轴的判别力。

### §5.1.1 Outcome-supervised Evaluator Internalization

Outcome-supervised evaluator internalization 训练 Evaluator 对完整候选行为给出全局判断，把一个完整 response、trajectory 或 episode 映射为 outcome-level 的评价。其吸引力在于 outcome-level label 通常比 dense process label 更易获得，且对具有明确最终答案或显式成功标准的任务可直接对齐最终目标。结构性局限是 credit assignment 弱：一条失败轨迹可能含有许多有价值的中间决策，一条成功轨迹也可能含有偶然有效但并不稳健的推理，把整条轨迹压成单一标签会掩盖哪些动作应被强化、哪些应被修正。

#### §5.1.1.1 判别式 outcome 评估器

判别式 outcome 评估器的产物是一个针对完整候选的标量分或离散标签，下游消费路径最短。

具身领域的 episode-level success/failure detector 构成最集中的一簇。Vision-Language Models as Success Detectors 把 success detection 重写为 visual question answering，用人工标注的 clip-level 成功标签对 Flamingo 的视觉相关层做 SFT，学习对「视频片段 + 任务描述」回答 yes/no 的能力。I-FailSense 面向跨任务 failure detection 分两阶段训练——先对 VLM 做 LoRA SFT，再训练附着在中间层的 FailSense blocks，用 cross-entropy 与 BCE 把 expert demonstration 与目标错配负例的 success/failure 信号内化为 detector。SAFE 把输入从纯观测扩展到 VLA policy 的内部 latent embedding，用 MLP 的 L1 目标或 LSTM 的 BCE 目标学习 trajectory-level 失败预测；其训练标签只有整条轨迹成败这一粗粒度信号，归 outcome，但推理时可逐时刻输出 failure likelihood 并配合 conformal threshold 触发 abort 或 human intervention。RoboReward 把 OXE demonstration 与带人工核验进度分的 RoboArena rollouts 汇成经验，经 counterfactual relabeling 与 temporal clipping 构造 1–5 级 episode rubric，对 Qwen3-VL 的 fusion 与 LLM 层做 SFT、以 MAE 回归离散 episodic reward。

判别式 outcome 也涵盖 trajectory-level preference model 与 web 域评估器。VLP 不依赖人工偏好，从 Meta-World 的 expert/medium/random 三类操作视频段出发，用 Intra-Task、Inter-Language、Inter-Video 三类隐式 preference 规则构造比较标签，以 cross-entropy 学习一个 trajectory-segment 级跨模态 preference model，产物是单个 scalar preference score，供 offline RL 做 reward annotation。WebRL 在自进化在线课程 RL 框架中训练 outcome-supervised reward model，用 WebArena-Lite 的 web rollout 及在线新生成的 trajectory buffer，依据用户指令、历史动作与最终网页状态判定整条 web trajectory 是否完成。WebRL 同时还学习了一个 value network critic——但该 critic 由 rollout return 与 GAE 更新、与当前 policy 强耦合、随策略迭代失效，属于策略优化的训练脚手架而非可独立复用的 Evaluator artifact；真正构成稳定 P4 产物、可跨 episode 复用的是那个 outcome RM。POLICYGUARD 把 WebArena trajectory 连同 domain metadata 构成 POLICYGUARDBENCH，对 Qwen3-4B 做 full-parameter SFT，把 policy violation detection 训练成严格格式的二分类指令跟随任务；它对完整轨迹与不同长度前缀都做判定，这是同一个 violation detector 被施加到不同输入长度上，而非两套不同粒度的监督，故仍归 outcome。

#### §5.1.1.2 生成式 outcome 评估器

生成式 outcome 评估器对完整候选生成 verdict 之外的语义内容——judging rationale、failure reasoning、repair suggestion——产物信息密度更高，但推理与输出开销也更大。

reasoning 与 general 域的 generative reward model / verifier 是这一子类的主线。Generative Reward Models 把 reward modeling 改写为 next-token generation，用 UltraFeedback、UltraInteract 的 preference pair 与自生成 rationale，训练直接 SFT 的 GenRM、带推理链的 CoT-GenRM，以及用 STaR 自举 rationale 的 SFT/DPO 变体——direct 变体对外只暴露 verdict token、可读为标量，CoT 变体输出 rationale 加 verdict。Generative Verifiers 同样把 reward modeling 当 next-token prediction，用 SFT 学 Yes/No verdict，并以 unified loss 联合 solution generation 与 verification，GenRM-CoT 先给验证推理再给 verdict，下游用于 Best-of-N reranking 与 weighted self-consistency。这两项工作落在判别式与生成式的交界处：其 direct 变体确实只产出可塌缩为标量的 verdict，归入生成式子类是因为方法贡献与 CoT 变体不可切分、且 next-token 形式本身允许产物携带 verification reasoning。S2J 指出 generative reward model 存在 solving 与 judging 能力的落差，要求 Evaluator 判断前先自行求解，用 solving reward 与 judging reward 组成联合 RL 目标（DAPO/GRPO）更新参数。J4R 用 EIS-GRPO 训练 pairwise judge，通过 response order swap 等 equivalent initial state 子组上的 judgment correctness 驱动 RL，压低位置偏置与表面格式干扰。Critic-RM（Self-Generated Critiques）让模型自生成 critique 作为 latent rationale，以联合目标同时优化 critique generation loss 与 Bradley-Terry preference loss，产物兼有连续 reward 与解释性 critique。AgentV-RL 把 verifier 做成多轮 tool-augmented 验证 agent——先做 rejection fine-tuning 学高质量 verifier 轨迹，再用 GRPO 以最终 verdict 与 ground truth 的一致性为奖励，产物是含代码执行的多轮 verification 过程与 True/False 结论。

另一组生成式 outcome 评估器以「让反馈帮助下游修复」为训练目标。CTRL（Teaching Language Models to Critique via RL）面向 code，把问题、候选程序、critique 与修复后程序表现组成经验，以 sandbox 隐藏单测的通过情况为奖励——先用 execution-guided synthetic critique 做 SFT，再用 GRPO 直接优化 critique 使其更能帮助 generator 产出通过测试的代码，产物是 analysis、actionable suggestion 与 correctness judgment。RL4F 训练 critique generator，以 refined output 相对 ground truth 的任务指标为 PPO 奖励，产物是反馈文本，回送给黑盒 task model 做输出修复。ECHO（No More Stale Feedback）用 dual-track GRPO 协同进化 critic 与 policy——critic 对完整 trajectory 生成高层 diagnostic critique，其奖励来自 critique 引导修正后 refined trajectory 相对 baseline 的性能提升，故训练信号本身挂在整条 refined trajectory 的最终表现上、归 outcome。

具身与 computer-use 域也出现了生成式 outcome 评估器。CriticGPT（Enhancing Robotic Manipulation with AI Feedback）从 Meta-World 不同训练阶段策略的操作视频出发，用环境全信息脚本自动生成 pairwise analysis 与 preference，对 LLaVA-1.5 做 LoRA 微调学习视觉比较与文字分析；这里被经验直接更新的核心 Evaluator 是 CriticGPT 本身，其下游再蒸馏出的 scalar reward model 属于 utilization 环节。ARMOR（Self-Refining VLM）以大规模 sparse failure label 与较小规模 dense reasoning annotation 训练机器人失败检测器，联合优化 BCE detection head 与 next-token reasoning decoder，并在 online refinement 阶段吸收自身 rollout，产物同时含 binary failure label 与自然语言原因分析。Video-Based Reward Modeling for Computer-Use Agents 构建 ExeVR-53k，对 Qwen3-VL 做 reward modeling 微调并引入时空 token pruning 处理长视频，联合学习 holistic success judgment 与 first-failure attribution——主任务是 video-level success、归 outcome，而 first-failure attribution 以时间区间形式给出结构化定位，使产物超出纯标量。

**§5.1.1 讨论。** outcome 子类内部，判别式与生成式评估器呈现不同的 utilization 画像。判别式 outcome 评估器（success detector、episodic RM、trajectory preference model）下游消费路径最短、延迟最低，适合 reward-filtered behavior cloning、offline RL annotation 与运行时 guardrail。生成式 outcome 评估器以更高的推理与输出开销换取可解释性与可修正性：其 verdict 仍可塌缩为标量进入 P6，但 rationale/critique 部分支持的是 critique-driven refinement 这类用法——这类用法在单篇论文里往往表现为「critique 驱动一次 revision 再把修正轨迹拿去训练」的链式结构，在 §7 作为 composite 处理。

### §5.1.2 Process-supervised Evaluator Internalization

Process-supervised evaluator internalization 训练 Evaluator 对中间推理步骤、动作前缀或单个 decision 提供局部监督。核心动机是 agent 失败通常不是最后一步的突然事件，而是一系列局部错误的累积；process 评估器试图回答 outcome 监督抹平的问题——不仅判断最终 outcome，还判断每个中间决策是否正确、是否有希望、是否推动任务进展。

#### §5.1.2.1 判别式 process 评估器

判别式 process 评估器对每个 step、prefix 或 state-action 对输出标量分，是这一支中规模最大的一簇。

reasoning PRM 谱系奠定了局部监督的基本范式并不断细化其语义。Let's Verify Step by Step 用 PRM800K 的人工 step 标注训练 PRM，对每一步给 positive/negative/neutral 判断，产物是每步 scalar correctness probability。OmegaPRM 用 MCTS 加 binary search 自动定位 first error，把前缀的可完成性转成 soft process label 以替代人工标注，并比较 soft pointwise、hard pointwise 与 pairwise Bradley-Terry 三类目标。Rewarding Progress 把监督目标从 step correctness 改写为 step-level advantage——从各前缀展开 Monte Carlo rollout 估计 future success probability，离散成 buckets 训练 Process Advantage Verifier，评估该步是否真正提高未来成功概率。PQM 把 PRM 训练从独立分类改为带 margin 的 comparative Q-value ranking，要求正确前缀的成功概率高于错误前缀、错误处价值出现显著断裂。Error Typing 引入 hierarchical supervision，两阶段 token prediction 先预测 math error 与 consistency error 两层错误类型、再预测最终 correctness reward。BiRM 联合学习 backward-looking process reward 与 forward-looking value 两个头，同时编码「当前步是否合理」与「该前缀未来是否还有希望」。

agentic、GUI 与 web 域把上述抽象信号落地为与环境状态变化直接对齐的 step-level 评估。AgentPRM 把 reasoning PRM 推广到多步 agent 决策，用 TD 与 GAE 从 rollout 自动构造 step-wise Q 值与 advantage 伪标签，以联合 MSE 同时学习 promise 与 progress 两个头。ProgRM 通过 LCS 匹配 recipe 的自标注算法在 GUI 轨迹中定位 key step 并分配归一化 progress label，对 Progress Reward Model 做 BCE 训练、预测当前动作执行后在任务 recipe 中的进展位置。Building Autonomous GUI Navigation via Agentic-Q Estimation 把终局 success 信号反传到 state-thought-action 三元组，用 BCE 训练 step-wise success probability estimator，再以 SWPO 配合 GRPO/RLOO/REINFORCE++ 做策略优化。Advancing Mobile GUI Agents 采用 Pairwise Process Preference——每一步让正确动作分数高于若干错误动作——以 Q-LoRA 微调 backbone 与评分头。IntentScore 把 planning intent 作为条件信息融入 action encoder，用 contrastive alignment 与 margin ranking 学习「界面状态 + 候选动作意图」的单步评估，以极低延迟服务 test-time reranking。GAIA 用 data flywheel 迭代训练 step-level binary critic——先 SFT 学第一轮 critic，再让其挖掘更难样本训练第二轮，持续纳入真实 GUI agent 的错误分布。code 域的 SWE-Shepherd 把读文件、编辑、跑测试等中间动作用启发式赋即时奖励、折扣累积成 step-wise target，以 qLoRA 加 MSE 训练 PRM。

具身领域的若干视觉 reward model 同样落在判别式 process。RL-VLM-F 让 GPT-4V/Gemini 对 observation image pair 先做 analysis 再给 pairwise label，用 Bradley-Terry 式 preference learning 学习连续 reward function；其监督落在状态或短段比较上而非整条 episode verdict。LIV 结合 VIP 风格的时序 value pretraining 与 CLIP 风格 InfoNCE，对初始帧、中间帧、目标帧之间的相对位置施加学习信号，在无动作标注时形成 dense reward。Video-Language Critic 以对比式 symmetric cross-entropy 加 sequential ranking objective 学习，强制成功视频中后期片段分数高于前期片段，对视频前缀输出 scalar reward；其对比目标针对整段视频与指令匹配、带 outcome 成分，但产物逐时刻、监督含明确的局部时序排序约束，归入 process。

#### §5.1.2.2 生成式 process 评估器

生成式 process 评估器对每个中间步骤生成 critique、错误诊断或修正建议，不止判断某步好坏，还解释违反了什么约束、错误属于何种类型、应朝什么方向修正。

reasoning 域的 generative PRM 在 step-level critique 上做深。DeepCritic 用 Monte Carlo 估计每步后的成功率并定位 first error，分 critique teaching 的 SFT 与 critique incentivization 的 GRPO 两阶段，产物是长文本 deliberate rationale、每步正负 judgment 与 first-error index。Enhancing LLM Reasoning via Critique Models 构建 MathCritique-76k——GSM8K 与 MATH 上受控生成的 flawed reasoning path 加 GPT-4o 对首个错误步骤的 critique，经 soft filtering 保留真正有助修正的样本——对 critique model 做 SFT 输出 step-level label 与 constructive feedback，同时服务 test-time conditional refinement 与 training-time 探索数据改写。StepWiser 把 NuminaMath-CoT 的中间 reasoning chunk 与其 Monte Carlo rollout 按前后 Q-value 变化标注 positive/negative，用 GRPO 直接强化 judge 生成正确 stepwise verdict 与解释，使其学习 meta-reasoning。

GUI 与 web 域是生成式 process 评估器最密集的领域。GUI-Shepherd 把 AndroidWorld 在线 rollout 与 AndroidControl 离线状态汇成 52k 交互经验，人工给每步动作打正负标签、GPT-4o 补解释性 rationale，对 UI-TARS-1.5-7B 做 SFT 学 step-level verification，并研究 CoT verifier 变体把推理与判定一并学进参数，产物既可是 positive/negative logit、也可是 rationale 加判定。GUI-Critic-R1 是 pre-operative critic——在动作执行前生成 observation analysis、possible result、critique、0–1 correctness score 与 corrective suggestion——先做 RFT cold-start，再用 Suggestion-aware GRPO 强化 correctness、reasoning 与 suggestion 质量。OS-Oracle 整合七个开源 GUI trajectory 数据集成 31 万 step-level 样本，规则模拟四类常见错误动作作负例，先 SFT 学 reason 加 judgment、再用 CP-GRPO 以 accuracy、format 与 reasoning-consistency 奖励强化跨平台 critic。WebArbiter 是 principle-guided generative PRM——用 WEBPRM COLLECTION 约 3 万 step-level preference pair，先用 o3 教师推理做 reasoning distillation、再用 GRPO 对齐最终 verdict 与 ground-truth correctness，产物含 principle induction、structured justification 与 preference verdict。Web-Shepherd 把 web process evaluation 做成 checklist-grounded generative judging——以 next-token prediction 让模型给定 instruction、checklist 与 state-action pair 时生成 feedback 文本与各 checklist item 的 Yes/No/In-Progress 判断，并可经 verbalizer 转成 scalar reward。Don't Act Blindly 训练 action-effect 验证器——负例通过制造「动作后界面无变化」的假历史构造——先做 robust SFT、再用 Verification-Action-Effect GRPO 以 action quality、effect honesty 与 verification accuracy 组成的复合奖励训练，产物是带 Verify/Diagnose/Recall 标签的 reasoning 与 SUCCESS/NO_CHANGE verdict。

具身领域的生成式 process 评估器把逐时刻进度判断与文字分析结合。AHA 通过 FailGen 从 RLBench 成功 demonstration 程序化生成失败子任务，按 LLaVA 指令微调范式做 autoregressive next-token 训练，让模型对长程操作中的中间 subtask 同时输出 yes/no 判断与自由文本 failure rationale，可用于 reward 改写、TAMP 规划修正与 zero-shot subtask verification。SOLE-R1 把真实与模拟机器人视频转为 120 万条 spatiotemporal CoT trace，用几何距离、时间反转与偏离注入自动得到逐时刻 progress 监督，先 SFT 学结构化 reasoning 加 progress value、再用 RLVR 与 GRPO 优化进度判断精度，产物是 think 区的时空分析文本加 answer 区的整数 progress score。Large Reward Models 对同一 Qwen3-VL backbone 用 LoRA、SFT、DPO 分别学习 contrastive reward、absolute progress reward 与 completion reward，把 24 个来源视频的时序经验内化进统一 LRM——其中 progress 与 contrastive 为 process 监督、completion 为 outcome 监督，产物为 reasoning-first 分析文本加 reward label。general agentic 场景下，PPRM（Hybrid Reward Normalization）面向不可验证的多轮 search 型任务，用 GPT-4o 与 Qwen3-235B-A22B 按 correctness、relevance、consistency 等原则给 step-wise 分数，对 PPRM 做 SFT 学生成分析文本与标准化分数，策略优化时以 ReNorm 把 process reward 与 outcome reward 混合。

**§5.1.2 讨论。** process 子类的演进有清晰脉络：早期工作先在 reasoning 中确立 step-level evaluator 的必要性，随后把局部监督从 correctness 扩展为 advantage、Q-value、progress 与双向 reward 语义，再在 GUI、computer-use、code 与 embodied 中落地为与环境状态变化、任务推进直接对齐的 Evaluator。process 评估器的固有代价是 step 级监督的获取——人工 step 标注昂贵，自动方案（MCTS rollout、binary search 定位 first error、TD/GAE 伪标签、LCS recipe 匹配）本质是用计算换标注，且这些伪标签的可靠性直接限制了 Evaluator 的上界。判别式与生成式 process 评估器之间，生成式一支信息密度最高，适合错误类型复杂、任务空间开放的场景，也有助于跨任务迁移——「未验证工具返回结果」「忽略用户约束」「过早停止搜索」等诊断可能在不同任务中重复出现。其代价是可靠性：自然语言 critique 可能听起来合理却与环境真实反馈不一致，refinement suggestion 可能误导下游；在高风险任务中，生成式 process 评估器通常需要与环境验证、程序执行或 human audit 联用。

产物形态与下游 utilization 自由度之间存在单调关系——生成式产物信息含量最高，下游可塌缩成标量进入 P6 的 RL、可取 binary 做 Best-of-N 筛选、也可保留全文驱动 step-wise refinement，utilization 自由度最大；判别式产物则下游选择最受限，只能作为数值信号被消费。这一关系在 §8 与 P6（Evaluator-to-Policy）的对接中进一步展开。
