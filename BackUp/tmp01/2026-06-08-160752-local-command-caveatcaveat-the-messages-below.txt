  二、Utilization 分析框架：第一性推导

  我不从 §2.4 的四维出发，也不从那个模型回复的具体建议出发。我从 agent
  的决策规则出发，问一个最基本的问题：

  ▎ 经验在推理时如何影响 agent 的决策 $a_t \sim \pi_\theta(\cdot \mid c_t)$？

  这条决策规则把 agent 的决策依据分置于两处：流经 $c_t$ 的外部信息，与固化在
  $\theta$ 中的内部能力。经验要影响决策，必须通过这两处之一进入决策循环。以此为
  起点，推导过程如下。

  Layer 0：消费接口——载体结构锁死的耦合

  Carrier 的存在形式决定了它能以什么方式进入决策循环。这不是选择，是结构约束：

  消费接口: Context Injection
  进入点: 经验写入 $c_t$，经前向传播影响 $a_t$
  对应载体: Narrative、非可执行的 Schematic
  ────────────────────────────────────────
  消费接口: External Execution
  进入点: 经验在模型外执行，结果写入 $c_t$ 或直接产生动作
  对应载体: 可执行的 Schematic
  ────────────────────────────────────────
  消费接口: Forward-Pass Computation (attention)
  进入点: 经验以连续向量参与 attention/hidden-state 计算
  对应载体: Latent
  ────────────────────────────────────────
  消费接口: Forward-Pass Computation (weights)
  进入点: 经验固化在权重中，前向传播自动激活
  对应载体: Policy
  ────────────────────────────────────────
  消费接口: Signal Provision
  进入点: 经验提供评估信号，用于 scoring / ranking / selection
  对应载体: Evaluator

  这五种接口是穷尽的——决策规则只有 $c_t$ 和 $\theta$ 两个入口，经验要么走外部（$
  c_t$），要么走内部（$\theta$），要么走一个外部评估回路（Evaluator 信号不进
  $c_t$ 也不进 $\theta$，而是作用在决策后的 selection/ranking 上）。

  这个映射的分析价值： "选载体"同时锁定了"怎么用"。你要可执行验证 → 必须走
  Schematic；你要零边际推理成本 → 必须走 Policy。这是 utilization-driven
  selection 的硬约束层。

  Layer 1：五个消费维度——所有 carrier 可统一刻画

  现在的问题是：给定一种 carrier
  的消费接口已被锁定，用什么维度来刻画它的消费特性，使得：
  - 每个维度对所有 carrier 都有意义（没有 "N/A" 列），
  - 维度之间互不冗余，
  - 能够区分 carrier 选择以支撑 §8 的场景推荐？

  以下五个维度满足这三个条件。我从"决策时需要什么"反向推导每个维度的必要性。

  ---
  Dimension 1: Consumption Interface（消费接口）

  - 问题：经验从架构的哪个位置进入决策循环？
  - 推导：这是 Layer 0 的直接产物，是所有后续维度的前提。把它作为第一列，读表的
  人先看到"这个载体本质上是怎么被用的"。
  - 取值空间：Context Injection / External Execution / Forward-Pass (attention)
  / Forward-Pass (weights) / Signal Provision

  Dimension 2: Activation Selectivity（激活选择性）

  - 问题：能否在推理时有选择地激活特定经验，还是整个载体始终"全开"？
  - 推导：决策时不是所有经验都相关。Agent 需要的是"与当前 $c_t$
  相关的那些经验"，而非全部。不同载体在这个能力上差异悬殊，且这个差异直接决定
  carrier 在"精确复用 vs. 广泛泛化"谱系上的位置。
  - 取值空间：
    - Per-item：可以精确选择单条经验（Narrative：retrieve specific memory）
    - Per-skill / Per-node：按功能单元选择（Schematic：invoke specific skill）
    - Per-bank / Per-session：按粗粒度块选择（Latent：load specific latent
  bank）
    - Indivisible / 
  Always-on：不可分割，始终激活（Policy：所有经验融合在权重中）
    - Per-evaluator：选择用哪个评估器，但评估器内部不可分割（Evaluator）

  Dimension 3: Cost Structure（成本结构）

  - 问题：使用经验需要支付什么资源，何时支付？
  - 推导：不同 carrier
  的成本发生在不同阶段、由不同资源承担。必须拆成两个子维度，因为 fixed cost 和
  per-use cost 的承担者不同，优化的方向也不同：
    - Fixed cost（固定成本）：使经验可用的一次性投入（storage + indexing /
  curation + testing / training / fine-tuning）
    - Per-use cost（单次使用成本）：每次复用经验时支付的代价（context tokens +
  retrieval latency / execution compute / attention integration / zero marginal
  / forward pass）
    - Where work happens（计算发生位置）：GPU context processing / external
  executor / "free" in weights。这三个位置对应完全不同的资源瓶颈。
  - 为什么这个维度必须有：carrier 选择的核心 trade-off 就是 fixed cost vs.
  per-use cost。Narrative 固定成本低（存文本），每次复用都付 token 成本；Policy
  固定成本高（需训练），每次复用零边际。没有这一列，§8 的 cost-driven selection
  没有数据。

  Dimension 4: Control Freedom（控制自由度）

  - 问题：在推理时，系统设计者可以在不重新生产 carrier 的前提下调节什么？
  - 推导：这是 carrier 之间最不对称的维度，也是 §8 场景推荐的软约束层。Narrative
  可以调检索策略、top-k、排序、截断、prompt 格式——几乎每个环节都可控。Policy
  几乎为零——权重一旦训练完，在推理时你无法选择"用哪部分经验、不用哪部分"。这个不
  对称直接映射到 interpretability、correctability、adaptability 三个下游需求。
  - 取值空间：High / Medium / Low / Near-zero，并标注具体可调维度（retrieval /
  orchestration / invocation / integration / none）

  Dimension 5: Maintenance Mechanism（维护机制）

  - 问题：新经验持续到来时，载体如何演化？
  - 推导：这是 utilization 与 maintenance 的桥梁。不同 carrier
  的更新方式完全不同——Narrative 可以逐条增删改，Policy
  必须重新训练。这一列点到机制类型即可，深入分析归到 verifiability →
  correctability → maintainability 结构规律（你们上一轮讨论的那条链）。
  - 取值空间：Append/Edit/Delete / Version/Replace / Retrain/Reload / Fine-tune
  / Replace evaluator

  ---
  Layer 2：从维度到选择——§8 的消费驱动推荐

  Layer 1 的五个维度构成一个坐标系。给定任务约束：
  - 延迟敏感 → 排除 per-use 支付 context token 的载体 → Narrative 出局 → 考虑
  Policy 或 Latent

  先确认分析对象。一条 pathway $\mathcal{T}: \mathcal{C}{\text{src}} \rightarrow
  \mathcal{C}{\text{tgt}}$
  是一个转化操作——把源载体中的经验重新表示为目标载体。Pathway 比较不是在比两个
  carrier 谁好（那是 §2.3
  已经做完的），而是在比从源到目标的这条转化路径本身有什么 trade-off。

  这意味着：
  - 目标载体的属性不是 pathway 的属性。 "Policy 可解释性低"是 Policy carrier
  的属性，不是 Policy Internalization pathway 的专属特征——Evaluator
  Internalization 也产出低可解释性的 Parametric 载体。把 carrier 属性当 pathway
  属性来比，等于把 §2.3 复读一遍。
  - Pathway 独有的东西是转化过程本身。 同一份 trajectory，走 Narrative
  Abstraction 变成 reflection 和走 Policy Internalization
  变成权重更新，两者的差异不仅在最终载体不同，更在转化过程的代价、信息损失、可验
  证性、增量能力上完全不同。

  所以 pathway comparison
  的分析对象是转化操作的结构特性，而非目标载体的静态属性。

  ---
  一、从转化操作的结构推导维度

  把一条 pathway 看作一个函数 $f: \mathcal{E}{\text{src}} \rightarrow 
  \mathcal{E}{\text{tgt}}$，其中 $\mathcal{E}{\text{src}}$ 
  是源经验集合，$\mathcal{E}{\text{tgt}}$
  是转化产物。这个函数可以从三个截面被刻画：

  截面 A：输入侧——"喂什么进去"

  转化操作对输入有要求。不是任何 trajectory 都能喂进任何
  pathway——有些需要成功轨迹，有些需要对比对，有些需要人类标注。

  A1. Source Carrier（源载体类型）
  - 直接来自 pathway 定义。7 条路径源端分属 Tokenized (P1–P5) 和 Parametric
  (P6–P7) 两类。
  - 这一列是恒定的，不需要"比较"，但需要标出以建立表的行身份。

  A2. Data Prerequisite（数据前提）
  - Volume：需要多少源经验？单条 trajectory 够不够（Reflexion 式的
  reflection），还是必须跨轨迹归纳（multi-trajectory induction），还是需要大规模
  batch（SFT/RL）？
  - Quality signal：源经验需要带什么监督标签？只需 trajectory
  本身（无监督），需要 success/failure 信号（弱监督），需要 step-level
  annotation（过程监督），需要 pairwise preference（偏好监督），还是需要 human
  label（强监督）？
  - Source diversity：对经验来源有要求吗？只能用自己的
  trajectory（self-generated），还是可以用 human demo / teacher model？

  A3. Input Robustness（输入鲁棒性）
  - 转化操作对噪声输入有多敏感？失败轨迹、含噪声的动作序列、不完整的
  observation——这些还能用吗，还是会导致转化失败或产物质量骤降？
  - 这一维把 "需要什么样的经验" 与 "经验质量不够时会怎样" 区分开。

  ---
  截面 B：转化过程——"转化本身怎么发生、代价多大"

  这是 pathway comparison 的核心——载体属性管不到这一层。

  B1. Transformation Semantics（转化语义）

  转化操作对经验做了什么性质的重新表示？这是最基础的分类维度，直接决定 pathway
  的"信息命运"。穷举七条路径：

  ┌──────────────┬──────────────────────────────────────────────────────────┐
  │     路径     │                         转化语义                         │
  ├──────────────┼──────────────────────────────────────────────────────────┤
  │ P1 Narrative │ 语义抽象：从具体轨迹提炼规则/反思/insight，保留语义内容  │
  │  Abstraction │ ，提升抽象层级                                           │
  ├──────────────┼──────────────────────────────────────────────────────────┤
  │ P2 Schematic │ 结构形式化：从自然语言轨迹抽取可执行/可遍历的结构，注入  │
  │  Formalizati │ 形式约束                                                 │
  │ on           │                                                          │
  ├──────────────┼──────────────────────────────────────────────────────────┤
  │ P3           │                                                          │
  │ Latent-Space │ 连续压缩：离散 token →                                   │
  │  Transformat │ 连续向量，保留计算状态，丢弃符号可读性                   │
  │ ion          │                                                          │
  ├──────────────┼──────────────────────────────────────────────────────────┤
  │ P4 Evaluator │                                                          │
  │  Internaliza │ 评估能力内化：交互经验 → 判断函数，学会"评价"而非"执行"  │
  │ tion         │                                                          │
  ├──────────────┼──────────────────────────────────────────────────────────┤
  │ P5 Policy    │                                                          │
  │ Internalizat │ 决策能力内化：交互经验 → 生成函数，学会"执行"而非"评价"  │
  │ ion          │                                                          │
  ├──────────────┼──────────────────────────────────────────────────────────┤
  │ P6 Evaluator │ 偏好迁移：评估能力 → 决策偏好，Evaluator 的判断转化为    │
  │ -Driven      │ Policy 的生成倾向                                        │
  │ Optimization │                                                          │
  ├──────────────┼──────────────────────────────────────────────────────────┤
  │ P7           │                                                          │
  │ Parametric   │ 知识外化：隐式权重 → 显式                                │
  │ Externalizat │ token，不可读的能力重新变成可读的数据                    │
  │ ion          │                                                          │
  └──────────────┴──────────────────────────────────────────────────────────┘

  这七种语义不是标签，而是信息转换的类型学——每一类定义了经验被重新表示时信息的组
  织方式。后文的 Information Dynamics 就是在此基础上展开。

  B2. Production Mechanism（生产机制）

  转化靠什么计算来执行？
  - LLM inference（prompting）：Narrative Abstraction、Schematic Formalization
  的主流机制——模型读 trajectory，输出 artifact。不需要训练，每次转化付 inference
  cost。
  - Gradient-based training：Policy Internalization (SFT/RL)、Evaluator
  Internalization、Latent（Prompt-Based/Module-Based）、Evaluator-Driven
  Optimization——需要 GPU 训练。
  - Gradient-free optimization：Cache-Based Latent（直接存 KV）、部分
  Schematic（parsing + template filling）——不需要反向传播。
  - Execution + verification：Programmatic Skill（生成代码 → 执行 → 验证 →
  入库）——转化过程包含环境反馈闭环。

  这个维度的分析价值在于：它决定了你需要什么基础设施。 只需要 API key 就能做
  prompting 式转化；需要 GPU 集群才能做 training 式转化。

  B3. Production Cost（生产成本）

  拆为三个子维度：

  - Computational cost：一次转化需要多少计算？数量级差异巨大——LLM inference
  一次几秒 + $0.01，SFT 一批几 GPU-hour，RLHF/DPO 完整 pipeline 几十到几百
  GPU-hour。
  - Human cost：需要人类参与吗？P4 (RM training) 需要 human preference
  label；P1/P2 可以全自动。Human-in-the-loop 不是 binary，而是 continuum。
  - Cost amortization：成本是一次性的还是每次新增经验都要重新支付？这是
  Incrementality（B5）的成本面。

  B4. Information Dynamics（信息动态）

  这是 pathway comparison 最核心的分析维度，也是 carrier analysis（§2.3,
  §8.2）完全覆盖不到的。转化操作同时做三件事：保留一些信息，丢失一些信息，生成一
  些新信息。三个子维度：

  - Information 
  preservation（信息保留）：源经验中的什么语义内容在转化后仍可获取？
    - P1：核心洞察与模式被保留，具体步骤和时序细节被抽象掉
    - P2：过程结构被保留，自然语言中的隐含假设和上下文 nuance 可能丢失
    - P3：模型的内部计算状态被保留（token 无法还原的 hidden
  representation），但符号内容不可直接读取
    - P5：决策模式被保留在权重中，单条经验的贡献不可区分
    - P6：评估偏好被保留，但评估器的原始校准信息丢失
    - P7：权重中的"能力"通过采样的方式部分恢复为 token，但采样有偏且不完整
  - Information loss（信息损失）：转化丢弃了什么？损失的是噪声还是信号？
    - 关键区分：lossy-by-design（抽象、压缩——损失细节但保留结构）vs.
  lossy-by-limitation（训练不充分、采样偏差——损失可能包含信号）
    - P1/P2 的损失是 by-design（抽象必然丢弃细节）
    - P5 的损失可能是 by-limitation（SFT 欠拟合、RL reward
  hacking）——这不是设计意图，是工程风险
  - Information gain（信息增益）：转化是否产生了源经验中不存在的新信息？
    - P1：跨轨迹归纳可能产生单条轨迹中不存在的综合 insight
    - P2：形式结构（如 workflow DAG 中的依赖边）可能在原始 log 中只是隐式存在
    - P3：连续表示可能捕获离散 token 未显式编码的统计规律
    - P6：Evaluator 信号可能引导 Policy 发现单靠 trajectory 无法学到的行为模式
    - P7：外化过程依赖采样，可能产生源权重中并不存在的 hallucination
  - Traceability（可追溯性）：转化产物的元素能追溯到源经验的哪个部分吗？
    - P1：高——reflection 引用的失败模式可对应到具体 trajectory 步骤
    - P2：中——workflow 步骤可大致对应但经过泛化
    - P3：低——连续向量无法逐元素解释
    - P4/P5：极低——权重更新是 batch 梯度累积的结果，不可追溯单条经验
    - P6：极低——Evaluator 信号的来源已在 P4 阶段丢失
    - P7：中——生成的 trajectory 是新的，不可追溯权重的具体成分

  Information Dynamics 这一列是整个 pathway comparison
  分析增量最大的地方——它不重复 §2.3 的载体属性，也不抢 §8.2
  的消费分析，而是填上了"转化过程中经验发生了什么"这个独属于 pathway
  层面的问题。

  B5. Incrementality（增量性）

  新经验持续到来时，转化操作能否增量执行，还是必须从头重做？

  - Incremental：新 trajectory → 新 reflection 写入 store（P1 Static）；新
  trajectory → 生成新 skill 入库（P2 programmatic
  skill）——新经验只触发新条目的生成，不影响已有条目。
  - Incremental with side effects：新条目可能触发旧条目的 merge/prune（P1
  Dynamic store）——增量但有级联效应。
  - Batch reprocess：训练式转化（P3 Prompt-Based/Module-Based、P5
  SFT）——增加新数据需要重新训练，但可以从旧 checkpoint 热启动。
  - Online/iterative：RL-based 转化（P5
  RL、P6）——可以在线更新，但分布漂移使旧经验的效果衰减。
  - Full retrain：P4 RM training with human
  labels——新增人类标注通常需要重新训练以保持校准。

  B6. Output Verifiability（产物可验证性）

  转化完成后，能否独立验证产物是否正确？注意这不是 carrier 的
  interpretability（"我能不能读 Policy 权重"），而是
  "我能不能验证这次转化操作本身是否成功"。

  - Direct inspection：P1 的 reflection 可以直接阅读判断是否合理——人类或强 LLM
  都可做
  - Execution test：P2 的 code skill 可以跑——通过/失败是客观信号
  - Behavioral test：P5 的策略只能通过 rollout
  结果间接判断——新策略比旧策略好吗？好多少？不确定
  - No direct verification：P4 Evaluator 的质量只能通过它在下游 P6
  中的效用间接评估；P3 Latent 表示的质量只能通过 ablation 间接判断
  - Circular verification risk：P6 用 Evaluator 训练 Policy，再用 Policy
  的行为评估 Evaluator——存在循环验证风险

  这一列直接连接你们之前讨论的 verifiability → correctability → maintainability
  结构规律——如果转化产物不可验证，就不可纠正，进而不可维护。Pathway
  选择时，如果任务需要长期维护，应避开 Output Verifiability 为 "no direct" 或
  "circular" 的 pathway。

  ---
  截面 C：输出侧——"产出的东西能接着干什么"

  输出侧属性大部分是目标 carrier 的属性（§2.3 已覆盖，§8.2
  将展开），但有一个维度是 pathway 特有的：

  C1. Composability（可组合性）

  这个 pathway 的产物能否直接作为另一个 pathway 的输入？这是 §7 composite
  pipelines 的基础。

  - P1 (Narrative) → 可作为 P2 (Schematic)、P3 (Latent)、P4 (Evaluator)、P5
  (Policy) 的输入
  - P2 (Schematic) → 可作为 P5 (Policy) 的输入（skill as action
  space）；不太直接进 P3/P4
  - P4 (Evaluator) → 唯一用途是进 P6（或作为 P5 的 reward shaper）
  - P5 (Policy) → 可作为 P7 (Externalization) 的输入，也可作为 P1 的输入（policy
  rollout → trajectory → ...）
  - P7 (Tokenized) → 可作为任何以 Tokenized 为源的 pathway 的输入——P7 是整个
  pathway 图的闭环节点

  Composability 的分析价值在于：它揭示了哪些 pathway 是"终端"（P4
  Evaluator——到了 Evaluator 基本就到头了，只能进 P6），哪些是"枢纽"（P1
  Narrative——几乎可以接任何下游 pathway），哪些是"闭环"（P7 + P5
  构成自生成环）。

  四、§8.1 的呈现结构

  不把 7 条 path × 7
  维全部铺平成一张大表——那会变成读不懂的密集矩阵。更好的做法是分组 + 一张总表：

  分组逻辑

  按转化语义的"跨度"将 7 条路径分为三组，每组内部有共享的 trade-off 模式：

  Group 1: Same-Level Refinement（P1, P2）
  - 共同特征：源和目标都在 Tokenized 层内，转化靠 LLM
  inference，保持可读性，可增量
  - 对比焦点：抽象（P1）vs. 形式化（P2）的信息动态差异——P1
  损失细节但保留语义灵活性，P2 注入结构但牺牲自然语言的 nuance

  Group 2: Cross-Level Internalization（P3, P4, P5）
  - 共同特征：从 Tokenized 跨到
  Latent/Parametric，转化靠训练，可解释性断崖式下降，增量性受限
  - 对比焦点：压缩程度与信息损失的 trade-off——P3 保留计算状态但最轻量，P4
  专化评估能力但终端化，P5 直接改变决策但不可追溯

  Group 3: Parametric-to-X（P6, P7）
  - 共同特征：以 Parametric 为源端，都是"用已有的隐式经验做第二件事"
  - 对比焦点：P6 是"从评估到生成"的参数间迁移（进权重），P7 是"从权重到
  token"的外化（出权重）——方向相反，Composability 角色互补（P6 终端，P7 闭环）

  总表

  三组 prose 分析之后，附一张精简总表。表不追求穷尽所有维度，只保留跨组差异最显
  著的几个维度：

  行 = 7 条路径，列 = Transformation Semantics / Production Mechanism /
  Information Loss Level / Traceability / Incrementality / Output Verifiability
  / Composability。

  五、与 §8.2 和 §8.3 的衔接

  §8.1 Pathway Trade-off Comparison
        ↓ 输出：每条 pathway 的生产代价画像
  §8.2 Carrier Utilization Analysis
        ↓ 输出：每种 carrier 的消费代价画像
  §8.3 Utilization-Driven Pathway Selection
        ↓ 输入：生产画像 + 消费画像 → 场景匹配

  衔接点在 §8.3。举例：
  - §8.1 告诉你：P1 (Narrative Abstraction) 生产成本低（LLM
  inference），可增量，可验证；P5 (Policy Internalization)
  生产成本高（SFT/RL），需 batch retrain，不可追溯。
  - §8.2 告诉你：Narrative carrier 每次复用付 token 成本，控制自由度高；Policy
  carrier 零边际成本，控制自由度近乎零。
  - §8.3 据此给出场景推荐：如果需要频繁更新经验且要求可审计 → P1 →
  Narrative；如果经验体量极大、延迟要求极严 → P5 → Policy。

  §8.3
  不引入新数据，只做交叉匹配。这正是它不应该是一个长章节的原因——它的全部价值来自
  §8.1 和 §8.2 的分析质量。

  六、一个写的时候要警觉的坑

  的文献综述，颗粒度在"这类方法怎么做"。§8.1 是 pathway-level

  7 条路径之上，用上述 7 个维度做横向对比，用具体工作做 illustration 而非
  enumeration。每一句话都在比，而不是在列。

  ---
  一、从第一性原理推导框架

  起点：Pathway Selection 到底在选什么

  回到 agent 决策规则：

  $$a_t \sim \pi_\theta(\cdot \mid c_t)$$

  这条规则的工程含义是：经验要影响决策，必须通过两个入口之一——要么进入
  $c_t$（外部、显式），要么进入 $\theta$（内部、隐式）。一条 pathway
  的价值就在于它把经验送到了哪个入口、以什么形式、付了什么代价。

  所以 pathway selection 的本质不是"选哪条路"，而是"经验应该放在架构的哪个位置，
  以什么形式存在，付多少生产/消费代价"。Pathway 只是到达那个位置的路径——先选目标
  位置（carrier），再选路径（pathway），再检验路径是否走得通。

  这给出了选择逻辑的三阶段结构：

  约束分析 → 目标载体选择 → 生产路径选择 → 可行性检验

  而不是扁平化的 "约束 → 路径" 直接映射。

  ---
  Step 1: 约束从哪里来

  任务场景对 agent 系统施加的约束，按来源分为四个不可互相归约的类别：

  Class 1: 推理时资源约束（Inference Resource Constraints）

  这些约束来自部署环境，不来自任务语义。

  ┌─────────────┬────────────────────────────┬──────────────────────────────┐
  │   子维度    │            含义            │        为什么不可归约        │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ Latency     │ 单步决策允许的最大延迟     │ 实时对话 vs.                 │
  │ budget      │                            │ 离线批处理，差三个数量级     │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ Context     │ 上下文窗口的可用空间       │ 长程任务 vs. 短问答，可用    │
  │ budget      │                            │ context 量悬殊               │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ Compute     │ 每步可用的 GPU/CPU         │ 手机端 vs.                   │
  │ budget      │                            │ 云端，算力天花板不同         │
  ├─────────────┼────────────────────────────┼──────────────────────────────┤
  │ Per-call    │                            │ 百万级 API 调用 vs.          │
  │ cost        │ 每次推理允许的最大金钱成本 │ 单次研究实验                 │
  │ ceiling     │                            │                              │
  └─────────────┴────────────────────────────┴──────────────────────────────┘

  Class 2: 经验规模约束（Experience Scale Constraints）

  这些约束来自经验的积累速度和总量。

  ┌───────────────────┬──────────────────────┐
  │      子维度       │         含义         │
  ├───────────────────┼──────────────────────┤
  │ Stock             │ 已有多少历史经验     │
  ├───────────────────┼──────────────────────┤
  │ Velocity          │ 新经验以多快速度到达 │
  ├───────────────────┼──────────────────────┤
  │ Diversity         │ 经验跨多少任务/领域  │
  ├───────────────────┼──────────────────────┤
  │ Retention horizon │ 经验需要保留多久     │
  └───────────────────┴──────────────────────┘

  Class 3: 质量与治理约束（Quality & Governance Constraints）

  这些约束来自组织/监管/安全需求，不来自技术限制。

  ┌────────────────────┬────────────────────────────────┐
  │       子维度       │              含义              │
  ├────────────────────┼────────────────────────────────┤
  │ Interpretability   │ 人类是否需要审查经验的使用过程 │
  ├────────────────────┼────────────────────────────────┤
  │ Correctability     │ 错误经验需要多快能被修正       │
  ├────────────────────┼────────────────────────────────┤
  │ Safety criticality │ 错误决策的后果是否不可逆       │
  ├────────────────────┼────────────────────────────────┤
  │ Audit trail        │ 是否需要事后追溯决策依据       │
  └────────────────────┴────────────────────────────────┘

  Class 4: 环境动态约束（Environment Dynamics Constraints）

  这些约束来自外部世界的变化速度。

  ┌───────────────────────────┬────────────────────────┐
  │          子维度           │          含义          │
  ├───────────────────────────┼────────────────────────┤
  │ Task distribution drift   │ 任务本身是否随时间变化 │
  ├───────────────────────────┼────────────────────────┤
  │ Interface stability       │ 工具/API 是否稳定      │
  ├───────────────────────────┼────────────────────────┤
  │ Experience staleness rate │ 旧经验多快失效         │
  └───────────────────────────┴────────────────────────┘

  ---
  Step 2: 约束如何转化为载体压力

  这是 §8.3 的分析引擎——整套选择逻辑的核心。我对每个约束维度，从 carrier
  的结构属性出发，推导它施加的方向性压力。

  推导遵循一个统一的逻辑：约束 → 需求 → 哪些 carrier 满足/违背该需求 → 
  方向性压力（toward / against）。

  以下是完整的压力映射。注意：这是 carrier-level 的压力，不是 pathway-level
  的——先确定哪些载体能活下来，再考虑怎么生产它们（pathway feasibility）。

  ---
  2.1 Latency Sensitivity（高延迟敏感）

  需求：每步决策的总时间极短。经验载体引入的推理时开销必须最小化。

  逐载体分析：

  - Narrative：需要 retrieval（搜索外部 store）+ context
  processing（模型处理注入的 token）。检索延迟 + 上下文处理的 prefill
  时间直接叠加在决策延迟上。检索规模越大延迟越高。→ 强 against
  - Schematic (executable)：外部执行需要调用
  executor/API/runtime，执行时间不可控（可能秒级），且可能失败触发重试。→
  against（但弱于 Narrative，因为执行可以异步？实际不然——agent
  通常需要执行结果才能继续决策，所以也是同步阻塞）
  - Schematic (non-executable, injected into context)：同 Narrative，需 context
  processing。→ against
  - Latent：attention integration
  在前向传播中完成，不增加独立的检索/执行步骤，开销与常规 forward pass 融合。→
  toward（条件是 latent 检索开销低于 token 检索 + 编码）
  - Policy：零额外开销——经验在权重里，前向传播自动激活，不需要检索、不需要额外
  context token。→ 强 toward
  - Evaluator：需要额外的 forward
  pass（评估候选/对输出打分），增加延迟。但如果只用于 offline selection（如
  best-of-N 只做一次），延迟可以摊销。作为实时 gate（每步评估并决定是否
  abort），则是纯增量延迟。→ 弱 against（取决于部署方式）

  压力方向：Policy >> Latent > Evaluator (offline) > Schematic > Narrative

  ---
  2.2 Experience Volume（大量经验）

  需求：经验体量大时，每次决策都能有效利用相关经验，且成本不随体量失控。

  逐载体分析：

  - Narrative：检索质量随 store 增大下降（召回精度降低），context budget
  限制单次可注入的经验数量。体量越大，要么牺牲覆盖率（只检索
  top-k），要么牺牲精度（放宽阈值）。→ against
  - Schematic：skill library 增大后 selection 变难（选哪个 skill），但每个 skill
  独立执行，不像 Narrative 那样争夺 context budget。体量对单次使用影响较小。→
  弱 toward
  - Latent：latent memory bank 体量受 memory 容量和检索机制限制。Module-based
  latent 的 memory 过大会导致读写效率下降。→ 中性偏弱 against
  - Policy：训练数据体量不影响推理成本（O(1)），经验被"压缩"进固定大小的权重。但
  训练成本随体量增长。→ 强 toward（消费侧），但有生产侧代价
  - Evaluator：训练数据体量不影响推理时的单次 forward pass 成本。→
  toward（消费侧），同样有生产侧代价

  压力方向：Policy > Evaluator > Schematic > Latent > Narrative

  ---
  2.3 Interpretability Requirement（强制可解释）

  需求：人类必须能够审查"什么经验被使用了、为什么被使用、经验内容是什么"。

  逐载体分析：

  - Narrative：人类可直接阅读注入 context 的经验内容（reflection、rule
  等），可审计检索结果。→ 强 toward
  - Schematic：结构可审查（workflow DAG、code
  skill），但语义理解需要技术能力。执行结果可观测。→ toward（弱于 Narrative）
  - Latent：连续向量不可直接阅读。最多通过 probing / 解码回 token
  近似还原，但这不是真正的可解释。→ against
  - Policy：权重完全不可读。无法解释"哪段经验导致了当前决策"。→ 强 against
  - Evaluator：可以看 evaluator 输出的分数/critique，但无法解释打分依据如何从训
  练数据中得来。评判逻辑封装在权重中。→ weak against（比 Policy
  略好，因为至少能看到输出信号）

  压力方向：Narrative >> Schematic > Evaluator > Latent > Policy

  ---
  2.4 Correctability Requirement（快速可修正）

  需求：当发现经验有误（错误规则、过时 skill、有害
  bias），需要能快速定位并修正，不影响其他正确经验。

  逐载体分析：

  - Narrative：逐条编辑/删除。修正一条 reflection 不影响其他条目。→ 强 toward
  - Schematic：skill/workflow 独立版本控制。修正一个 skill 重新测试后替换。→
  toward（修正成本比 Narrative 高——需重新测试，但仍然是局部的）
  - Latent：无法定位单条经验在 latent space
  的贡献。修正需要修改训练数据后重新训练整个 latent bank。→ against
  - Policy：无法定位单条经验在权重中的贡献。修正需要重新训练（SFT/RL），且可能破
  坏其他能力（catastrophic forgetting）。→ 强 against
  - Evaluator：同 Policy——重新训练以修正 bias，可能破坏校准。→ 强 against

  压力方向：Narrative > Schematic >> Latent > Policy ≈ Evaluator

  ---
  2.5 Environment Volatility（环境高变动）

  需求：环境/任务/工具快速变化时，经验载体必须能快速适应，旧经验不能成为负资产。

  逐载体分析：

  - Narrative：可以快速追加新经验，标记/删除过时条目。适应速度取决于 store
  的维护机制（Static 需人工清理，Dynamic 有自动淘汰）。→ toward
  - Schematic (executable)：执行接口变化导致 skill 直接不可用（API 变了，code
  跑不起来）。需要重新生成 + 测试。→ against（因为对 interface stability 敏感）
  - Latent (session-scoped, 如 KV cache)：天然无 staleness 问题——每次 session
  重新产生，旧 session 自动失效。→ 弱 toward
  - Latent (trained, 如 soft prompts)：需重新训练。→ against
  - Policy：需重新训练以适应新分布。训练期间旧 policy
  持续退化。训练成本高、周期长。→ 强 against
  - Evaluator：同 Policy——需重新训练以适应新任务评估标准。→ 强 against

  压力方向：Narrative > Latent (session) >> Schematic > Latent (trained) >
  Policy ≈ Evaluator

  ---
  2.6 Safety Criticality（极端安全敏感）

  需求：错误决策的后果不可逆。经验在被使用前必须能被验证。

  逐载体分析：

  - Narrative：使用前可阅读验证（"这条 reflection 说得对吗？"）。可设置人工审核
  gate。→ 强 toward
  - Schematic：使用前可测试验证（skill 在 sandbox 跑 → 看结果对不对）。可设置
  test gate。→ 强 toward
  - Latent：无法事前验证——只能看 latent 注入后的行为变化，无法验证 latent
  本身。→ against
  - Policy：无法事前验证——只能看 rollout 结果。→ 强 against
  - Evaluator：可做 calibration check（在已知 ground truth 的 hold-out set
  上评估），但无法穷尽验证所有输入。→ 弱 toward（比 Policy 好——至少 evaluator
  的准确性可部分验证）

  压力方向：Narrative ≈ Schematic > Evaluator > Latent > Policy

  ---
  Step 3: 多约束解析——独裁、联盟与张力

  单约束分析给出的是每个维度上的方向性压力。真实的场景选择面对的是多约束同时作用
  。这些约束之间有三种逻辑关系：

  3.1 独裁约束（Dictator Constraint）

  某些约束是 hard constraint——不满足就直接排除载体，不与其他约束"加权平均"。

  独裁约束的典型来源：
  - Safety criticality = extreme → Policy 和 Latent
  直接出局，无论延迟和体量多有利
  - Interpretability = mandatory（监管要求） → Policy 和 Latent 直接出局
  - Latency budget < 50ms（硬实时） → Narrative 和 Schematic-execution 直接出局

  独裁约束的作用是缩小搜索空间——先排除，再在幸存者中优化。

  3.2 联盟约束（Aligned Constraints）

  多个约束指向同一方向时，形成联盟，推荐信号增强。联盟不是简单的"同方向相加"——它
  意味着即使某个约束在未来放松，推荐依然稳健。

  - 高延迟敏感 + 大经验体量 + 低可解释需求 → 三重 toward
  Policy。即使体量后来变小了，Policy 依然是最优
  - 强可解释 + 快速可修正 + 高环境变动 → 三重 toward
  Narrative。即使环境后来稳定了，Narrative 依然不差

  联盟约束的分析价值在于：它告诉你哪些推荐是
  overdetermined（过度决定的）——这些推荐的稳健性最高。

  3.3 竞争约束（Competing Constraints）→ 不可化约的张力

  这是 §8.3 最有分析增量的部分。当两个强约束指向相反方向时，单一载体无法同时满足
  ，必须通过多载体组合或分阶段策略来解。

  识别以下不可化约张力对：

  张力对: 低延迟 × 强可解释
  冲突: Policy 快但不透明，Narrative 透明但慢
  为什么不可化约: 没有载体同时满足两者——这是结构性的，不是工程可以优化的
  ────────────────────────────────────────
  张力对: 大体量 × 强可解释
  冲突: Policy 可 scale 但不可读，Narrative 可读但不可 scale
  为什么不可化约: 大体量下检索精度下降 + context 不够，无法靠增加检索预算解决
  ────────────────────────────────────────
  张力对: 高环境变动 × 低延迟
  冲突: Narrative 适应快但每次付 token 延迟，Policy 推理快但更新慢
  为什么不可化约: 环境变动要求频繁更新，低延迟要求零边际成本——自相矛盾
  ────────────────────────────────────────
  张力对: 快速可修正 × 大体量
  冲突: Narrative 可逐条修正但不 scale，Policy 可 scale 但修正需 retrain
  为什么不可化约: 大体量下逐条修正的人力不可承受，但 retrain 又太慢

  这些张力不是分析框架的失败——恰恰相反，它们是框架最重要的产出。它们解释了：
  - 为什么 §7 的 composite pipelines 存在（它们正是设计来部分缓解这些张力的）
  - 为什么没有普适最优载体（张力是结构性的，不是暂时的工程困难）
  - §9 的 open problems 该写什么（如何动态调节载体组合以响应张力变化）

  ---
  Step 4: 载体组合逻辑

  竞争约束的存在使载体组合不是可选项而是必需项。组合逻辑有三种模式：

  模式 A：并行互补（Parallel Complement）

  两个载体同时服务于同一决策，各自覆盖对方无法满足的约束。

  - 典型实例：Narrative + Policy。Policy 承担实时决策（满足低延迟），Narrative
  承担 audit trail（满足可解释）。Policy 出错时，Narrative
  中的反思提供纠错依据。
  - 文献对应：§7.2 Refinement-Mediated Policy Internalization 本质上就是先用
  Narrative 做监督准备，再转入 Policy 获得推理效率。

  模式 B：分层分级（Tiered Routing）

  按决策的重要性或难度分流到不同载体。

  - 典型实例：Policy (routine) + Evaluator (gate) + Narrative 
  (exception)。Policy 处理常规决策，Evaluator 检测异常/低置信度，触发
  Narrative-based 人工或 LLM 复核。
  - 文献对应：Evaluator 作为 safety gate 的模式散布在 §5.1 和 §6.1 中——几乎所有
  PRM/verifier 的工作都隐含这个路由逻辑。

  模式 C：时序分阶段（Temporal Staging）

  经验在时间维度上经历载体迁移：先以低成本载体快速可用，再逐步结晶为高效率载体。

  - 典型实例：Narrative (recent) → Policy (stable)。新经验先写成 reflection
  立即可用（低生产延迟），积累到一定量后经 P5 固化进 Policy（降低消费成本），旧
  reflection 可选择淘汰或保留为 audit trail。
  - 文献对应：§7.3 Generative Experience Curation
  的经验构造与筛选，以及自生成闭环中的 Policy → Tokenized → Policy 循环。

  ---
  Step 5: Pathway 可行性过滤

  载体组合确定后，对每个目标载体，检查能否负担生产这条载体的 pathway。这里接入
  §8.1 的数据：

  - 目标载体 Narrative → P1 (Narrative Abstraction) 成本低、可增量。可行。
  - 目标载体 Schematic → P2 (Schematic Formalization) 成本中（需要 LLM inference
  + 可能的执行验证），需要 interface 相对稳定。检查环境约束。
  - 目标载体 Policy → P5 (Policy Internalization) 成本高（SFT/RL），不可增量。需
  要检查：训练资源有吗？环境足够稳定以收回训练投资吗？
  - 目标载体 Latent → P3 (Latent-Space Transformation)
  成本中到高，取决于子类。Cache-based 便宜但不跨 session。Prompt-based /
  Module-based 需训练。
  - 目标载体 Evaluator → P4 (Evaluator Internalization)
  成本高，需要标注数据或明确的成功/失败信号。

  如果目标载体在消费侧完美匹配约束，但唯一能生产它的 pathway
  在生产侧代价不可承受 → 这是一个真实的困境，如实记录，指回 §9（需要更低成本的
  Policy Internalization 方法，或更高效的 Latent 训练）。

  三、§8.3 的呈现结构建议

  框架是分析逻辑，呈现是传达逻辑。二者不应混淆。§8.3 的读者需要先看到"这套选择是
  怎么做的"，再看到"典型场景下这套逻辑怎么用"。所以呈现结构是：

  §8.3.1 约束维度与载体压力

  - 一张 Constraint → Carrier Pressure 总表：行 = 6 个约束子维度（从四类中挑出最
  discrimative 的），列 = 5 类载体，单元格 = toward / neutral / against +
  一句话理由。
  - 这张表是 §8.3 的核心数据——后面所有分析都是对它的使用。
  - 标注哪些约束是 hard（可成为 dictator）。

  §8.3.2 约束交互：联盟、独裁与不可化约张力

  - 联盟约束：举例说明什么时候推荐是 overdetermined 的
  - 独裁约束：举一个 hard constraint 驱逐看似最优载体的例子
  - 不可化约张力：列出最重要的张力对，每个对说明为什么单载体解不存在、复合解的空
  间在哪
  - 这一小节直接为 §7（为什么 composite 存在）提供需求侧解释

  §8.3.3 载体组合模式

  - 三种组合模式的展开（并行互补 / 分层分级 / 时序分阶段）
  - 每种模式给出一个代表性实例（从 §3–§7 literature 中取，但只做 illustration
  不做 enumeration）
  - 讨论每种模式的适用条件与脆弱点

  §8.3.4 从选择到演化

  - 约束不是静态的：任务分布会漂移、经验体量会增长、安全要求会变化

  五、不该做的事

  1. 不要做决策树。 "If latency-sensitive → choose Policy."
  这种写法是工程博客，不是 CSUR
  survey。框架的作用是揭示分析的维度与张力，不是给读者一个自动化的选择算法。
  2. 不要列"每个场景的推荐路径"清单。
  场景无限多，列举注定不穷尽且随意。应该给出约束维度 →
  压力的分析逻辑，让读者自己应用到自己的场景。场景举例只做
  illustration，承认其非穷尽性。
  3. 不要回避张力。 竞争约束无解的情况是分析产出，不是分析缺陷。敢于说"在 X 和 Y
  同时很强的场景下，目前没有好的单载体或简单组合方案"——这正是 §9 要解决的问题。
  4. 不要把 pathway 当原子选择。 一个系统通常走多条 path。§8.3 应该从 carrier
  组合出发，再为每个 carrier 匹配 pathway，而不是从 7 条 path 中单选。

  零、当前框架的结构

  §8.1  生产画像    §8.2  消费画像
        ↘          ↙
         §8.3  选择逻辑

  这个结构干净、逻辑顺序正确——但它是比较性的（comparative），不是合成性的（synth
  etic）。比较性分析告诉你 A 和 B 哪里不同。合成性分析告诉你为什么 A 和 B
  必然不同，以及这些不同意味着什么。

  CSUR 的 §8 标题是 "Cross-Pathway Synthesis"，不是 "Cross-Pathway
  Comparison"。Synthesis 要求的不只是把差异列出来，而是找到产生这些差异的深层原
  则——从而使框架不仅能描述已有的 7 条路径和 5
  类载体，还能推理任何尚未被发明的转化形式。

  当前框架缺的正是这些生成性原则。

  ---
  一、根本问题：缺失生成性原则

  我从一个简单的问题出发：

  ▎ 为什么 Narrative carrier 的消费成本是按次支付的 context token，而 Policy 
  ▎ carrier 的消费成本是零边际？这不是巧合。为什么 Tokenized → Parametric 
  ▎ 的五条路径中，有三条需要 gradient-based training，而 Tokenized → Tokenized 
  ▎ 的两条只需要 LLM inference？这也不是巧合。

  这些 trade-off
  之间存在结构性耦合——多个维度上的差异源于同一个深层原因。识别这些深层原因是
  synthesis 的核心任务。

  从 carrier 分类轴出发，可以导出三条生成性原则。它们不是额外的分析维度——它们是
  维度背后的维度，解释了为什么 §8.1 和 §8.2 的表格长成它们该长的样子。

  原则一：存在层次原则（The Existence-Level Principle）

  ▎ Carrier 在 Tokenized → Latent → Parametric 
  ▎ 谱系上的位置，同时决定了它的消费接口、每用成本、控制自由度与可追溯性。这些维
  ▎ 度不是独立变化的——它们被存在层次单调地锁定。

  推导：

  - Tokenized 载体以离散符号存在于模型外部 → 消费接口必然是 context injection →
  每用成本包含 context processing → 但离散符号可被独立寻址、读写 →
  控制自由度高、可追溯性高。
  - Parametric 载体以连续权重存在于模型内部 → 消费接口必然是 forward pass
  computation → 每用成本为零边际 → 但权重不可被独立寻址 →
  控制自由度近乎零、可追溯性近乎零。
  - Latent 载体位于两者之间 → 消费接口是 forward pass (attention) →
  每用成本低但不是零 → 连续向量不可直接阅读 → 控制自由度与可追溯性居中。

  这意味着：在这个谱系上，任何 carrier 的 consumption profile 
  都可以从其存在层次被预测。 不需要逐 carrier
  枚举——存在层次给出了约束边界，具体的 carrier 只在边界内做微调。

  这个原则是 §8 最重要的合成性产出。它说：你在 §8.2
  表里看到的不是五个独立的数据点，而是一条单调曲线上的五个采样点。

  对 §8.3 的影响：利用驱动的选择在根子上是存在层次的选择——你首先决定经验应该放在
  谱系的哪个位置（外部可读写 vs. 内部零成本），然后在这个位置内选择具体的
  carrier 子类。§8.3 的 Phase
  2（硬约束剪枝）本质上就是在谱系上画一条不可逾越的线。

  原则二：形式化原则（The Formalization Principle）

  ▎ 在 Tokenized 层内，Narrative → Schematic 的形式化程度差异决定了经验是被理解
  ▎ 消费还是被机制性消费，进而决定了产物的可验证性模式。

  推导：

  - Narrative 通过 language / multimodal understanding 消费 →
  验证靠"读"（人类或强 LLM 判断语义是否正确）→ 可验证性高但主观。
  - Schematic 通过 parsing / execution / traversal 消费 → 验证靠"跑"（execution
  test 给出客观通过/失败信号）→ 可验证性也高但是客观——两者的验证是不同质的。

  这个原则说：同为 Tokenized、同为高可验证性，Narrative 的验证和 Schematic
  的验证覆盖不同的错误类型。Narrative
  验证覆盖"说的对不对"（语义正确性），Schematic 验证覆盖"跑不跑得通"（形式正确性
  + 环境一致性），但前者可能漏掉"说得好但做不到"的经验，后者可能漏掉"跑得通但方
  向错"的经验。

  对 §8.3 的影响：如果任务的安全关键性来自"不能做错事"（execution safety）→ 偏向
  Schematic（可执行测试）；如果任务的安全关键性来自"不能理解错"（semantic
  safety）→ 偏向 Narrative（人类可审查）。两者不是同一种"安全"。

  原则三：功能角色原则（The Functional-Role Principle）

  ▎ 在 Parametric 层内，Policy → Evaluator 的功能角色差异决定了载体在 pathway 
  ▎ 图中的拓扑位置——Policy 是枢纽节点（可以进任何下游 pathway），Evaluator 
  ▎ 是终端节点（只能进 P6）。

  推导：

  - Policy 产生 action → action 产生新的 trajectory → trajectory 可以进任何以
  Tokenized 为源的 pathway → Policy 通过 P7 外化，重新进入整个 pathway 图。
  - Evaluator 产生评估信号 → 评估信号只能用于 scoring / ranking / reward →
  如果不与 Policy 结合（P6），评估信号无处可去 → Evaluator 是终点站。

  这对 Composability 的影响是结构性的：Policy 是 pathway
  图中的强连通分量成员（可以通过 P7 → P5 循环回到自身），Evaluator
  是汇节点（只有入边 P4，只有出边
  P6）。这个图论性质不是经验观察，是功能角色定义直接蕴含的。

  ---
  三条原则的价值不在于它们分别说了什么——每一条单独看都像是 §2.3 的简单重述。它们
  的价值在于同时成立：存在层次、形式化程度、功能角色三个维度合起来，定义了
  carrier 空间中的一个三维坐标系，任何 carrier
  在三个轴上的坐标就决定了它的生产特性（§8.1）、消费特性（§8.2）和在 pathway
  图中的拓扑角色（§8.3 的 composability 分析）。

  这就是 synthesis：不是比 A 和 B 的差异，而是给出一个坐标系统，使所有载体的所有
  特性都可以从这个坐标系统中被推出。

  ---
  二、第二个缺失：Pathway → Carrier → Consumption 的耦合链

  ---
  三、第三个缺失：时间维度

  当前框架是静态的——假定约束 profile 给定且不变。但 §3-7 的文献反复展示了一个事
  实：经验管理是持续进行的。经验体量单调增长。任务分布会漂移。工具的 API
  会变化。这意味着 optimal carrier configuration 是时间的函数。

  时间维度的缺失导致三个具体问题：

  问题 1：initial vs. steady-state 的选择可能不同。
  系统刚启动时经验少，Narrative store 的检索精度高、context 开销小，P1
  完全够用。当经验积累到百万条时，检索质量下降、context budget 吃紧，需要迁移到
  Policy（P5）。但框架没有告诉读者什么时候应该从 P1 切换到 
  P5，以及切换的代价是什么。

  问题 2：切换本身有代价。 从 Narrative 迁移到 Policy 意味着做一次
  P5——这是一次性的训练成本。如果切换过早（经验还不够），训练出的 Policy
  不够好，切换后反而退化。切换的时机是一个 optimization
  problem，而框架没有给它提供分析工具。

  问题 3：复合路径的时序意义被低估了。 §7 的 composite pipelines 被当前框架当作"
  另一种生产路径"对待，但它们的深层意义是时序优化——Refinement-Mediated Policy
  Internalization（§7.2）本质上是在说"先保持经验在 Narrative
  形式（可读、可修正、立即可用），积累到足够量且质量达标后，再固化进
  Policy（零边际成本）。"复合路径不只是在空间上组合多条边，而是在时间上编排经验
  在不同载体中的停留。

  这意味着 §8.3.4（"从选择到演化"）不应只是一个桥接段落，而应该是 §8.3
  的一个完整子部分，用以下分析内容：

  - 约束的时序导数：哪些约束随时间收紧（体量 ↑、staleness
  ↑），哪些随时间放松（训练资源越来越便宜），哪些不变（延迟上限是物理极限）。
  - 迁移触发条件：什么时候应该把经验从一种载体迁移到另一种？例如 Narrative →
  Policy 的触发条件：经验体量超过检索精度阈值 AND 经验分布趋于稳定 AND
  训练成本可摊销。
  - 迁移代价：迁移不是免费的。P5 的固定训练成本需要在 Policy
  的消费侧节省中被收回。如果系统寿命不够长，迁移可能是净亏损。

  ---
  四、第四个缺失：失配分析

  当前框架假设约束 → 载体 → pathway
  的映射可以做对。但实践中约束可能被误判（你以为延迟要求很宽松，但上线后发现不是
  ），或者约束可能发生变化（监管突然要求可解释性）。

  框架应该告诉读者：如果选错了，会发生什么？错误的表现是什么？哪些错误是可恢复的
  ，哪些是不可逆的？

  这不是悲观主义——这是验证框架解释力的方式。一个好的分析框架应该能够预测失配的症
  状。如果框架正确，那么"在需要高可解释性的场景中使用了
  Policy"就应该表现出特定类型的失败（无法定位错误来源、修正一个错误时破坏其他能
  力）——而这些恰恰是文献中观察到的现象。

  失配分析可以组织为以下结构：

  失配类型: 可解释需求 × Policy
  症状: 无法审计决策、错误定位困难、修正引发灾难性遗忘
  可恢复性: 低（权重中的经验不可单独提取）
  恢复路径: P7 外化 → 审查 → 筛选后重新 P5，代价极高
  ────────────────────────────────────────
  失配类型: 低延迟需求 × Narrative
  症状: 检索延迟叠加、context 过长导致 prefill 超时
  可恢复性: 高
  恢复路径: 渐进迁移到 Latent 或 Policy
  ────────────────────────────────────────
  失配类型: 大体量 × Narrative
  症状: 检索精度下降、context 塞不下、幻觉引用
  可恢复性: 中
  恢复路径: 做 P5 迁移，但需重新训练
  ────────────────────────────────────────
  失配类型: 高环境变动 × Policy
  症状: 策略过时、无法快速适应新工具
  可恢复性: 低
  恢复路径: 需重新训练，训练期间系统持续退化
  ────────────────────────────────────────
  失配类型: 安全关键 × Latent
  症状: 无法验证 latent 中的经验质量
  可恢复性: 低
  恢复路径: Latent 无法审计，只能放弃并重建

  这张表不是给读者一个"不要犯错"的清单，而是：
  - 验证框架：预测的症状与文献中观察到的 failure mode 一致 → 框架有解释力。
  - 连接 §9：每种不可恢复的失配都是一个 open problem——"如何在需要 X
  的场景中安全地使用 Y 载体？"

  六、但是——克制

  上述改进让我再审视一个风险：§8 会不会变得太重？

  算一下篇幅。§8.1（生产侧）+ §8.2（消费侧）+
  §8.3（选择侧，含演化与失配），即使在克制写作下，§8 的总篇幅可能达到 15-20
  页（LaTeX 双栏）。相比 §3-7 中任何一个单章节的篇幅（各自 8-15 页），§8
  确实会成为全文最长的 section。

  但这不是问题——§8 本就应该是全文最长的 section。论文的结构决策已经说了 §8
  是"核心差异化贡献 2"，与
  §7（复合路径）并列。差异化贡献需要篇幅来兑现。关键在于每一页都在做 synthesis
  而非 enumeration——如果 §8 的每一段都在"比"而不是在"列"，篇幅就是合理的。

  如果必须压缩，优先级是：
  1. §8.3.5（失配分析）可以精简为 §8.3.4
  末尾的几个段落——不独立成节，但保留核心内容
  2. 三条生成性原则的开篇不能压——这是整个 §8 的脊梁
  3. Pathway-Carrier 耦合表不能压——这是 §8.3 Phase 5 的数据基础
