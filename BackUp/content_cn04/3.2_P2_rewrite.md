## §3.2 Narrative-to-Schematic Transformation（P2）

P2 将 Narrative Tokenized Agent Experience 转化为具有显式结构约束的 Schematic Tokenized artifact。源经验是 Agent 序贯决策中产生的具体轨迹——task execution log、reasoning trace、tool-use trajectory、web navigation history、GUI interaction trace 或 embodied execution record。转化的核心动作是把弱形式化的 Narrative experience 形式化为可被解析、执行、遍历、实例化或调度的结构化 artifact。典型目标载体包括 executable code skill、parameterized action program、workflow、DAG、hierarchical procedure、task tree、knowledge graph、state-transition graph 与 structured memory graph。

判定一项工作属于 P2 需同时满足三个条件：输入是具体 Agent trajectory 或 interaction log（区别于 P7 中模型凭参数知识直接生成 artifact）；输出是具有可解析、可执行或可遍历结构的 Schematic artifact；下游复用机制性地利用这种结构，而非仅让 LLM 自由阅读其文本表面。第三个条件是 P2 与 P1 的关键分界——结构是否在下游决策中发挥机制性作用。

**分类依据。** P2 内部子类划分与 P1/P2 判定均不以论文自报术语为准。文献中 skill、experience、memory、procedure、workflow 等词存在严重的同名异物与同物异名：Voyager 的 skill 是可执行 JavaScript 代码，ExpeL 的 insight 是自然语言规则，A-MEM 的 memory 是图节点；反过来，同一种"带步骤的可调用单元"在不同论文里被称作 skill、procedure 或 routine。归类基于两个可观测属性：artifact 的结构形态（自然语言文本 / 可解析可执行结构 / 关系图），以及下游消费机制（依赖 LLM 阅读理解，还是依赖 executor、parser、graph traversal）。论文原文术语仅在介绍工作时保留，不参与归类判定。

本文按转化后 Schematic artifact 在后续 Agent 决策中承担的主要语义功能分为三类：**Programmatic Skill Construction**（经验转化为可调用的动作单元）、**Procedural Workflow Induction**（经验转化为多步任务流程结构）与 **Structured Memory Graph Construction**（经验转化为环境、状态或交互关系的图结构）。对图状 artifact，按主要语义功能而非表面拓扑分类：若图主要编码任务执行顺序、步骤依赖、控制流或阶段结构，归入 Procedural Workflow Induction；若图主要编码实体、状态、空间关系、环境知识或交互历史，归入 Structured Memory Graph Construction。部分工作的 artifact 跨越两类语义功能，或其 P2 归属本身存疑，这些情形在 §3.2.4 集中讨论。

### §3.2.1 Programmatic Skill Construction

Programmatic Skill Construction 从具体 Agent trajectory 中构建可调用、可执行或可参数化的技能单元，使 Agent 在后续任务中直接调用该单元，而非从低层动作重新规划。源经验通常是成功或经筛选的执行轨迹，转化后的 artifact 表现为 code skill、parameterized action program 或 callable skill interface。这类 artifact 的功能是"动作压缩"或"动作空间扩展"：原本需多步低层操作完成的行为被抽象为一个高层可调用单元，通常可被 executor、runtime 或 browser automation framework 直接执行。

§3.2.1 与 §3.2.2 的划分不取决于 artifact 内部是否含多步过程或前后置条件——这两个特征在两类中都普遍存在（PSN、Skill-Pro 的 skill 与 §3.2.2 的 For25、Tan26 的 workflow 都带 precondition/postcondition）。判据是 artifact 在下游决策中的调度地位：Programmatic Skill 是一个整体可选、可启停、可执行的动作单元，Agent 在动作空间中将其作为单个高层动作选中并执行；Procedural Workflow 则是组织整类任务执行的流程结构，planner 在其阶段间导航、依据当前观测判断所处阶段。一个 skill 可以充当 workflow 中的一个节点。

**代表性工作。** Voyager [Wan23c] 在 Minecraft 中通过 iterative prompting 反复生成、执行、修正 action program，只有经环境反馈、执行报错与 self-verification 验证通过的程序才写入 skill library；库以 skill description 的 embedding 为索引、值为可执行的 Mineflayer JavaScript 程序，新任务按 task plan 检索 top-k skills 放入 prompt 供组合调用。ASI [Wan25d] 从 Agent 在线求解中被 evaluator 判为成功的 episode 出发，清洗掉报错步骤后由 induction module 诱导出 typed Python functions，并把原轨迹重写为调用新 skill 的版本再 re-execute——只有重写后仍能复现成功的 skill 才入库，把 trajectory abstraction 与 execution-grounded re-verification 绑定。SkillWeaver [Zhe25c] 让 web agent 在自动课程驱动下自主 practice，用 reward model 识别成功行为，把 state-action 序列提示 LLM 生成带 typed parameters 的 Playwright API，再以 unit test、静态分析与环境反馈持续 honing；生成的 skill 带 function signature、docstring 与 prerequisite state，作为高层 API 加入动作空间。WebXSkill [Wan26d] 的输入是更强 teacher model 在 WebArena 与 WebVoyager 上合成执行的 synthetic trace，从中抽取 action subsequence，抽象为带 typed parameters 与 step-level guidance 的 JSON skill，并按 URL pattern 组织为 skill graph；skill 既可在 grounded mode 作为原子调用整段执行，也可在 guided mode 作为分步说明注入 prompt——尽管内部含 step sequence，整体仍被当作一个可调用技能单元检索和使用。

Skill-Pro [Mi26] 的 artifact 形态是自然语言而非代码，但仍归入本类：它通过 hindsight attribution 从成功失败轨迹抽取 semantic gradient，经 PPO-style trust-region gate 验证 counterfactual advantage 后提出新 skill，产物是 activation condition、execution procedure、termination condition 三元组；下游 Agent 在 Skill-MDP 中先选中一个 skill，再由冻结 LLM 在该 skill 约束下生成 primitive action。它是一个整体可选、可启停的动作单元，这一调度地位而非代码形态决定了它属于 skill。

PSN [Shi26] 与 AutoRefine [Qiu26] 的 artifact 跨越 skill 与 workflow 两类功能。PSN 把 skill 表示为带 executable control flow、precondition、postcondition 与 CHILDREN 调用关系的 symbolic program，Agent 在 skill network 上做 backward-chaining 规划；它不是从轨迹一次性蒸馏 artifact 的最纯案例，而是用持续的 execution trace 驱动 skill network 的 failure credit assignment、局部补丁、冗余合并与在线 structural refactoring，属"trace-driven evolution"型。AutoRefine 跨成功失败轨迹做对比抽取，产物分为 Skill Pattern 与 Subagent Pattern——前者是带步骤的自然语言 guideline 或 code snippet，接近 skill；后者是带独立 system prompt、reasoning loop 与内部 memory 的 specialized subagent，经 hierarchical delegation 使用，功能上更接近 workflow 中的子任务调度。两篇的跨功能性在 §3.2.4 一并说明。

Programmatic Skill Construction 的优势在于可执行性、可验证性与可组合性：artifact 可通过运行结果判断有效性，也可在复杂任务中组合多个 skill。其代价是对环境接口稳定性高度敏感，skill 的泛化边界难以确定，skill library 持续增长后需配套 selection、deduplication、versioning 与 stale detection 机制——PSN 的 maturity-aware update gating 与 SkillWeaver 的 API selection 模块正是对此的回应。

### §3.2.2 Procedural Workflow Induction

Procedural Workflow Induction 从多步 Agent trajectory 中归纳可复用的任务流程结构：workflow、DAG、hierarchical procedure、task tree 或 stage-action hierarchy。这类方法抽象的不是单个可调用动作，而是完成一类任务所需的阶段划分、步骤顺序、依赖关系、控制流、条件分支与前后置状态。许多复杂任务的可复用经验体现在执行过程的组织方式中：成功轨迹常含稳定的阶段模式（先收集信息、再筛选候选、再验证约束、最后提交），失败轨迹则可能暴露步骤顺序或依赖检查的缺失。与 Programmatic Skill 相比，workflow 的粒度更高——一个 skill 可以是 workflow 的一个节点，一个 workflow 可调度多个 skill、工具或子任务。

**代表性工作。** Agent Workflow Memory [Wan24] 从 web agent 的 canonical experience 中抽取 recurring sub-routine，把实例值替换为占位符，产物是由 environment description、reasoning 与 action program 组成的 workflow trajectory；它支持离线（人类标注或模型合成经验）与在线（仅对 evaluator 判正的轨迹做归纳）两种模式，workflow 既可追加进 memory prompt 供参考，也可包装成高层函数加入动作空间。它的承载形式是自然语言文本，但步骤骨架明确且下游确实依该结构复用，是 P2 中以文本为载体的 workflow 案例。WorkflowGen [Wei26] 在 node 与 workflow 两个粒度抽取经验，把结构相同的轨迹聚类合并为 generalized workflow template，节点带 fixed/generatable 标记；推理时按相似度三级路由——高相似直接复用、中等相似只重写变量节点、低相似重新规划。Hierarchical Memory Tree [Tan26] 把成功 web 轨迹切成 subgoal segment，构造 intent / stage / action 三层树，stage 层带显式 pre/postcondition，叶层把 DOM 标识替换为可迁移的 semantic element description；Planner 依当前 observation 与 stage 条件对齐所处阶段，Actor 再把抽象动作 ground 到页面，专门解决 workflow mismatch。For25 [For25] 把轨迹分段为 atomic procedure（goal、precondition、action sequence、postcondition 四元组），反复共同带来成功的 procedure 序列进一步抽象为带 continue/skip/repeat/abort 控制逻辑的 meta-procedure；运行时用 Bayesian expected utility 排序选择，并在成功失败积累后做 contrastive refinement 与 reliability-based pruning。Meta-Agent-Workflow [Tan25b] 把最优 tool-utilization 轨迹映射为 tool invocation template，实例化参数后拼接成面向 Dify 平台的 JSON DSL workflow，执行报错时定位故障节点并由 LLM 生成 revision plan 重构对应 DSL block。

MOBIMEM [Liu25i] 的经验组织横跨 workflow 与 graph 两种结构，本文以其 workflow 成分为主归属于此：它把 mobile 执行经验与用户纠错蒸馏为带参数槽位的 multi-level experience template（高层 template 编码任务级 control flow，低层保存具体 UI step），由 Task Rewriter 匹配并填充槽位生成可执行方案。其 graph 成分（组织 user profile 概念关系的 DisGraph、缓存 prefix path 的 ActTree）在 §3.2.4 一并说明。MOBIMEM 是一个 self-evolution memory 系统，template synthesis、exception-driven updating 与 rollback-based stale detection 三者耦合，其贡献在组件整合而非任一单一转化，可作为 Composite（§7）的候选；本文给定单一主归属避免在 §3.2 三个子节重复展开。

A2Flow [Zha25c] 从 expert demonstration 与 task-resolution 数据出发，把 case 级示范转成 code-based workflow 与初始 operator，经 operator clustering 得到 task-aware abstract operator，再用 MCTS 在节点、边与 operator 构成的搜索空间中优化整体流程，产物是可直接执行的 directed workflow graph。它的源是人类示范（Project_Infos 允许的 {human} 来源），输出是显式 workflow，但 MCTS 优化使它偏向 workflow synthesis 多于纯粹的 experience induction，在本类中属边界。

FlowMind [Liu26] 采用 Execute-Summarize 两阶段：先让模型借工具完成任务并记录 execution trace，再在独立 summarize 阶段用 graph-building tool 抽取 essential step、dependency 与 control flow，构成带 sequencing、branching、iteration 的 workflow graph。它的转化（trace → 结构化 workflow graph）符合 P2 三条件，但产物主要服务于复现、调试、审计与后续自动化系统调用，而非作为 Agent 持续检索复用的 memory；若本类的范围限定为"可持续复用的 workflow memory"，FlowMind 是边界案例。

Procedural Workflow Induction 的优势在于可解释性与可控性：workflow artifact 更短、更抽象，显式表达步骤依赖与执行顺序，便于定位任务的关键阶段。主要挑战是抽象粒度控制——过于具体则难泛化，过于抽象则丢失关键上下文——以及 workflow drift：环境状态变化后原 workflow 可能失效，需配套 consistency checking、precondition verification 与更新机制，Hierarchical Memory Tree 的低置信回退与 WorkflowGen 的三级路由都是针对这一问题的设计。

### §3.2.3 Structured Memory Graph Construction

Structured Memory Graph Construction 把 Agent trajectory 中的实体、关系、状态变化、空间结构、事件依赖或交互历史转化为可检索、可遍历、可更新的图结构记忆。这类方法的目标不是生成可执行动作，也不是归纳任务流程，而是构建 Agent 对环境、任务世界、历史经验或多智能体交互关系的结构化表示。Agent 的许多经验更适合组织为关系结构：环境对象间有空间关系，动作改变状态，历史事件间有时间或因果联系，多智能体交互中存在角色、信息流与依赖。若这些关系仅以 raw log 或自然语言 note 存储，后续 Agent 难以做精确检索、局部更新与多跳推理。

**代表性工作。** AriGraph [Ano24] 在 text-game 中从每步 textual observation 抽取 semantic triplet 加入 semantic memory，同时把完整 observation 作为 episodic vertex 并与当步 triplet 相连，形成同时编码世界知识与情节历史的 knowledge graph world model；下游先做 semantic search 再沿 episodic link 取相关 past observation 写入 working memory，环境状态变化时删除被推翻的旧关系。G-Memory [Zha25] 面向多智能体系统，把协作中的 atomic utterance 组织为 interaction graph，把整次 query 实例化为 query node，再由 LLM 总结出 insight node，形成 interaction / query / insight 三层图；新 query 到来时做上下双向 traversal——向上取高层 insight、向下取稀疏化的成功协作片段——并按 agent 角色裁剪后注入各自上下文。A-MEM [Xu25b] 把每次交互转成带 keyword、tag、contextual description 与 embedding 的 atomic note，通过 link generation 在新旧 note 间建立语义关系，新经验还会触发对旧 note 的重写；检索时先向量召回 top-k note，再沿 linked memory 与 box 扩展取回相关 note。它编码的是 relation structure 而非流程结构，归入 graph 支；其 link 在下游既作为检索扩展路径也作为关系组织维度，与典型 environment graph 的差异在 §3.2.4 说明。MobileGPT [Lee23] 通过 Explore-Select-Derive 流程把 mobile app 的已完成任务转写为 task / sub-task / primitive action 层级，并把页面与子任务关系组织为 transition graph（节点为功能等价的 app page，边为可复用 sub-task，边内承载 action sequence）；新指令先做 recall 对齐到 memory 中的 task path，匹配成功则直接按图中结构执行。它的核心 artifact 是页面节点与子任务边的转移图，故归入 graph；边内部承载过程步骤，与 workflow 子类有交叉。

Environment Maps [Fen26] 与 BrainMem [Ma26] 的 artifact 跨越 graph 与其他功能。Environment Maps 把 browser trace、DOM 与 screenshot 编译为 JSON 环境地图，含 context node、parameterized action node、workflow order 与 tacit knowledge，Agent 通过 Read/Grep/Glob 文件接口查询；它的 workflow 字段与 sequential edge 真实编码流程顺序，但核心是环境与可行动作的结构化表示，故以 graph 为主归属。BrainMem 在 embodied 场景维护 Trajectory KG（action-state transition）与 Spatial KG（room-object 空间关系）双图，episode 结束后由 Experience Agent 把成功轨迹提炼为 generalizable pattern、失败轨迹经 guided retry 压缩为 symbolic guideline；双 KG 是 P2 的 graph artifact，而 episode 级 semantic guideline 是 P1 的 Narrative artifact——BrainMem 因此横跨 P1 与 P2，本文以其 graph 成分归入此处，其 guideline 蒸馏部分可在 §3.1 的 cross-trajectory induction 处交叉引用。

Structured Memory Graph 与 Procedural Workflow 的区别在图的语义功能：workflow graph 回答"任务应如何执行"，节点与边表示步骤顺序、控制流或任务依赖；memory graph 回答"环境中有什么、状态如何变化、历史经验间有什么关系"，节点与边表示实体、状态、事件、空间或交互关系。这类方法的优势是支持关系级别的记忆复用，图结构便于局部更新，适合 embodied agent、web agent 与多智能体系统的 long-horizon planning 与 state tracking。其困难是图抽取容易产生错误实体与错误关系，图记忆随时间增长出现 memory bloat、过时信息与 retrieval noise，通常需配套 graph update、conflict resolution、pruning 与 confidence estimation——AriGraph 的冲突关系删除与 G-Memory 的 graph sparsification 是典型机制。

### §3.2.4 Boundary Cases and Scope Clarification

§3.2.1–§3.2.3 内已标注若干跨语义功能的工作：PSN 与 AutoRefine 跨 skill 与 workflow（主归属 §3.2.1），MOBIMEM 跨 workflow 与 graph（主归属 §3.2.2），Environment Maps 与 BrainMem 跨 graph 与其他功能（主归属 §3.2.3）。这些工作均给定单一主归属并在其主子节内说明跨功能性，不在多个子节重复展开；§8 的 cross-pathway 比较以其主归属计入。其中 BrainMem 的 semantic guideline 蒸馏属 P1，是一篇真正横跨 P1/P2 的工作。

**Hybrid schematic-narrative artifact。** Agent KB [Tan25] 从多种 agent framework 的异构轨迹构建跨域经验库，经 human-guided abstraction 抽取统一的 experience unit $E=\langle\pi,\gamma,S,C\rangle$，其中 $\gamma$ 是 structured predicate、$S$ 是 action-reasoning pair；应用阶段经 Reason-Retrieve-Refine 把检索到的经验综合成 preliminary execution plan 与 targeted fix。它的 experience unit 同时含 schematic 成分（structured predicate、编号式 agent_planning）与 narrative 成分（原则性的 agent_experience 文本），不是干净的 SOP、DSL 或 DAG。它带 self-evolving memory（新条目加入、LLM ranker 去重、基于 recency/frequency/transfer success 的 eviction），按结构形态属弱 P2；若 §3.2.2 范围限定为显式强形式化 workflow artifact，Agent KB 作为 hybrid 边界文献处理更稳妥。这一类提示一个一般原则：artifact 表面带字段或编号不等于 Schematic——一些把轨迹总结为带编号 workflow、SOP 或 guideline 的工作，下游消费仍主要依赖 LLM 自然语言理解、无 parser 或 graph traversal 介入，这类工作归 P1。§3.1 中作为"偏程序化 skill consolidation"留在 P1 的 AutoSkill、SkillClaw、SkillX 即属此情形：其承载物虽近 typed skill library，但下游靠 LLM 理解消费，不进 P2。

**Non-agent / human-software-trace 来源。** 一批工作从人类使用软件的低层事件流构建 workflow，与 P2 主 taxonomy 的边界取决于源数据是否具备 LLM-based Agent 序贯决策的 experience loop。From Logs to Agents [Jo26] 从创意软件的 raw system trace（csv/JSON 事件流）重建 creative workflow DAG，用于 Markov 式过程分析与 process-aware agent 的上下文；In-Context Ensemble Learning [Xu24b] 从 WONDERBREAD 的 human demonstration video 聚合出 step-ordered SOP，目标是 workflow understanding 与 documentation；Workflow Graphs [Cha20] 从 instrumented Tinkercad 的人类 3D 建模 trace 构建 W-graph，且属 LLM 之前的工作。这三者的源数据是人类软件使用记录而非 LLM agent 的交互轨迹，目标偏向过程理解、文档化或群体策略分析，不纳入 P2 主 taxonomy；Cha20 的 W-graph 作为"人类轨迹 → workflow graph"的方法学前身、Jo26 与 Xu24b 作为 workflow 重建的外延参照，可在相关讨论中引用。需与之区分的是 Project_Infos 允许的 {human} 来源：Agent Workflow Memory 的离线人类标注经验、A2Flow 的 expert demonstration 仍属 P2，因为其示范本身具备决策过程语义与异构动作空间。判定排除与否，看的是源数据是否承载序贯决策语义，与来源是人类还是 agent 无关。

**Source-side ambiguity。** 某些工作生成 code、workflow 或 graph，但 artifact 并非从具体轨迹提炼，而是模型据任务描述、专家知识或参数化能力直接生成。这类更接近 P7 或一般的 program/workflow synthesis，不计入 P2。

**Composite pipeline。** 一些系统先把 trajectory 转为 Schematic artifact，再将其用于数据合成、SFT 或 RL。若论文核心贡献是从经验到参数的内化、Schematic artifact 仅作中间表示，这类工作归入 Composite（§7）或 Tokenized-to-Parametric pathway；P2 章节可引用它们说明 Schematic artifact 的中介作用，但不作为纯 P2 方法。
